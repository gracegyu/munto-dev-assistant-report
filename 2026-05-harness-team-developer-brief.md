# Munto AI 하네스 — 개발자 공유 브리핑

**대상**: 문토 **개발자·리드·코치**에게 현재 하네스(`munto-dev-assistant`)의 **문제점, 개선 방향, 교육/온보딩 안**을 빠르게 공유하기 위함.

**관련 문서**:
- **AS-IS 분석**: [2026-05-harness-AS-IS.md](./reports/2026-05-harness-AS-IS.md) — 현재 하네스 구조·비판.
- **TO-BE 프로세스 가이드**: [2026-05-harness-TO-BE.md](./reports/2026-05-harness-TO-BE.md) — 개선된 프로세스 다이어그램·단계별 사용법.
- **학습 가이드**: [2026-05-harness-learning-guide.md](./reports/2026-05-harness-learning-guide.md) — 학습 로드맵·준비도 평가.

**근거 레포**: `munto-dev-assistant` (`AGENTS.md`가 진입점).  
**기준 일자**: 2026-05-14 (이후 학습·팀 피드백에 따라 계속 보강; 레포 변경 시 동기화 권장).

### Spec이라는 단어를 팀에서 이렇게 쓸 때

SRS·원페이저 **텍스트만**으로 끝나지 않고, **`dev-chain-design` 결과물**(예: DBML, Swagger, Unit TCL)까지 **포함하여 상위설계가 갖춰졌을 때** 비로소 **Spec 완료**. 특히 **DBML·Swagger는 Spec에 포함되는 산출물**로서 **개발자 간 충분한 Peer Review**를 거친 뒤, **`dev-chain-*`(자동화된 Chain의 구현·검증 단계)** 로 진행한다는 것을 팀 규격으로 **확정**한다. 하네스의 Development Chain(설계 산출 후 구현·검증)과 같은 방향이다. TO-BE 프로세스 전체 도식·단계별 사용법은 [개선 프로세스 가이드](./reports/2026-05-harness-TO-BE.md) 참고.

## 1. 한 페이지 요약

- 하네스는 **`.agents/` 단일 원본 + Claude/Cursor/Codex 어댑터**로 스킬·규칙을 공유한다.
- **Development Chain**(팀 정의 상 **Spec**(문서·상위설계 **+ DBML·Swagger 등에 대한 개발자 Peer Review**) 준비 후 구현·검증)이 에이전트 행동 원칙에 박혀 있다.
- **강점**: 프로세스 스킬화, MCP·CLI 연결, 서브에이전트 계약, 대형 SRS용 spec 리뷰 PM 패턴 등.
- **주의점**: 입력이 빈약하면 특히 **스펙 품질이 허울만 함**이 나옴; **컨텍스트는 사람이 먼저 박제**해야 한다. **`munto-spec-writer`만으로 SRS를 통째로 뽑게 두면**(파일 차원에서 **단계 게이트 미강제**) AI 산출에 **매몰**되고 심하게 고민하지 않아도 된다고 느끼는 **게으름·바쁨 병렬** 현상까지 나기 쉬우므로, 팀 표준은 §4.3·§4.4·§4.5, 스킬 개편 검토는 §3.5.

## 2. 현재 인지 문제점 및 리스크

### 2.0 `munto-spec-writer`와 목표 프로세스의 간극

| 관점 | 내용 |
|------|------|
| **현재 스킬** | `munto-spec-writer` 는 **SRS 텍스트** 위주 작성에 치우쳐 있다. 팀에서는 **Spec = 문서 + 상위설계(DBML 등)** 까지이므로, 「SRS만 있으니 Spec 끝」으로 오인하기 쉬운 허점이 크다 (`dev-chain-design` 은 별 스킬). 사람이 먼저 **1.2 · 2.1 · 2.2**를 완결하라는 규약도 스킬에 **의무로 박혀 있지 않다**. |
| **팀이 지향하는 협업** | **사람이 핵심을 주도**하고 AI는 보조해야 한다는 전제와 정반대로 쓰이기 쉽다면, **양질 스펙·작성자 역량 모두 성장 어렵다**고 보는 게 타당하다(§4.5). |

### 2.1 하네스·도구 레벨

| 이슈 | 설명 |
|------|------|
| **어댑터 드리프트** | 래퍼가 가리키는 `.agents/` 경로가 깨져도 빌드는 안 돌므로 놓치기 쉽다 → `scripts/check-adapters.sh`를 **정기 또는 CI(PR)** 에 붙일 것을 권장. |
| **README ↔ 실파일 불일치** | 커맨드 목록 예시 등 문서 스냅샷 오래됨 가능 → 신뢰 소스는 `AGENTS.md` + 실 폴더. |
| **Cursor 한계** | 서브에이전트/Task 병렬이 없음 → `dev-chain-design`·`munto-spec-review` 설계 의도와 **품질 격차** 가능. 명시적 **프롬프트 템플릿**(체크리스트)으로 보완 필요. |
| **텍스트만의 한계** | 스킬/룰이 «반드시» 강제하는 것 아님; **린트·테스트·승인 게이트** 등과 병행해야 불변식 방어 가능. |

### 2.2 이미 처리된 예시

- 2026-05-14: Codex 쪽 **Unit TCL 작성자** 어댑터 이름 불일치(`tcl-writer` ↔ 실제 **`unit-tcl-writer`**)가 `check-adapters`로 드러났으며, 해당 날 **`unit-tcl-writer` 어댑터로 정합 완료**됨.

### 2.3 Spec 관련 산출 품질 (문서 SRS·원페이저 및 상위설계 PR)

| 증상 | 원인 성격 |
|------|-----------|
| 겉 보기 형식만 맞음 | 브리핑 입력 부족(**컨텍스트 기아**). 또는 **한 번에 풀 작성** 패턴으로 **앵커링·피상적 작성**까지 겹치는 경우(§2.0). |
| SRS 섹션 의미 왜곡 | 한편 **`document/spec-templates/SRS_v3.3_template.md`** 에 각 항목 설명(한글+영문 안내, 예: 2.6 가정·종속 관계 정의)이 있다 → **모델이 정확히 로드하면** 레퍼런스 자체는 충분히 강함. 다만 실행 시 **헤더만 따라가거나 잘린 컨텍스트**에서는 여전히 6장 NFR 등과 혼선 가능 → **프롬프트·워크플로에서 “해당 절 템플릿 Read” 명시**(§4.6). |
| 리뷰 통과했는데 방향 엇나감 | `munto-spec-review`는 **문서 내·표준 대비 검수** 중심; **조직 진실**은 검증 못함. **`design-consistency-reviewer`/`dbml-reviewer`** 등은 **설계 산출 정합성** 보조일 뿐, **개발자 Peer Review(DBML·Swagger)** 를 대체하지 않음(§4.3 항목 7). |

| 항목 | 내용 |
|------|------|
| **어댑터 CI** | PR마다 `bash scripts/check-adapters.sh`; 실패 시 머지 금지. |

### 3.2 단기 (2~6주)

| 항목 | 내용 |
|------|------|

| **Cursor 패리티 가이드** | 서브에이전트 부재 환경용 **설계/스펙 리뷰 프롬프트 템플릿**(멀티 턴·체크리스트). |

골든 시나리오(설계 입력→기대 목차)·MCP 정책 표·스킬 메타데이터 규격·외부 스킬 포맷 대응·관측 가능성 검토 등(상세 우선순위는 플래닝 회의 추천).

### 3.4 성공 지표 예시

- `check-adapters` PR 실패율 추적  
- 하네스 Audit 트렌드

### 3.5 검토 과제(AI 작성 스킬·SRS 프로세스) — **즉시 스킬 개편을 전제하지 않음**

아래는 팀 피드백을 바탕으로 한 **향후 과제 목록**(회의 후 우선순위 확정).

| 과제 | 설명 |
|------|------|
| **팀 규격 우선**(스킬 변경 없음 가능) | SRS **`1.2 Product Scope`, `2.1 Product Perspective`, `2.2 Overall System Configuration`** 를 사람이 **먼저** 합의 가능한 완결도까지 쓴 뒤, 나머지를 AI로 확장한다는 규격을 노션/Wiki·코드 리뷰 정책에 명문화하면 `munto-spec-writer` 현재 형태만으로도 **위험이 크게 줄어든다**. |
| **`munto-spec-writer`(`.agents/skills/common/docs/munto-spec-writer/SKILL.md`) 정렬** | 기본 진입을 「전 SRS 일괄 생성」 **치우침을 완화**: **블록별 작성**(예: 7장만)·**선행 결과물 존재 확인**(1.2/2.1/2.2 제공 여부) 등을 스킬 **앞단**에 두는 수정안 검토. |
| **점진적 SRS(예: 2~3 단계)** | 단계별로 사람 리뷰를 끼우고 각 단계마다 해당 템플릿 소절만 로드하게 나누는 방안. **장점**: 앵커링·품질 완충 시간 확보 — **비용**: 스킬 UX·예시 시나리오 정리 필요. **구현 순서 후보**(의견): (1) 팀 규약·프롬프트 템플릿으로 시범 적용 → (2) 만족도·사고 사례로 효과 확인 → (3) 스킬 본문 반영 여부 결정. |

## 4. 교육·온보딩에 넣으면 좋은 내용

아래를 **워크숍 또는 내부 위키 요약**(30~60분)으로 묶는 것을 제안한다.

### 4.1 하네스 필수 숙지

1. **`AGENTS.md`** 위치와 역할 — 스킬·규칙 목록과 **금지 규칙**(예: DB MCP 전 SSM).  
2. **멀티 루트 워크스페이스** — `workspace/*.code-workspace`로 하네스 + 제품 레포를 같은 창에서 연다는 이유.  
3. **스킬 vs 슬래시 커맨드** — 대부분 자연어+스킬; 커맨드는 권한 제한 등 예외적으로만.  
4. **Development Chain** — 상단 «Spec이라는 단어…» 절(**Spec 완료 = 문서·상위설계 + DBML·Swagger Peer Review** 후 `dev-chain-*`) — AGENTS 규약.  

### 4.2 플랫폼 차이 한 줄 요약

- **Claude / Codex**: 서브에이전트 패턴 활용 가능(스킬에 정의된 PM·위임).  
- **Cursor**: 규칙(룰)·파일 편집 중심; **병렬 위임 없음** → 템플릿으로 순서 재현 교육.  

### 4.3 스펙 작성 권장 프로세스(팀 규격 권장)

1. **SRS «뼈대» 우선(레거시 표준 재확인, AI 무관하게도 동일)**: 사람이 **`1.2 Product Scope`, `2.1 Product Perspective`, `2.2 Overall System Configuration`** 를 **먼저** 합의·검증 가능할 정도까지 완성한 뒤, 나머지 절에 **살을 붙인다**(AI 포함). **`1.2` 서술 요건·예시 함정은 §4.4 및 `document/spec-standard.md`** 를 본다. 세 절이 비어 있는 상태에서 「SRS 전체 써줘」는 규격 밖 행위로 본다.  
2. **사람 고정 레이어**: 전략, Goal/Mission 범위, 메인 아키텍처(경계·핵심 연동 원칙) — 필요 시 브리핑 문서 한 장과 같이 제공.  
3. **최소 브리핑 문서**(노션 1p 등): 문제, 범위/비범위, 성공 지표, 제약, 레퍼런스 링크. 없으면 **`munto-spec-writer` 기대치를 낮출 것** 또는 단계 진행 보류.  
4. **작성 모드**: 기본값은 **사람 핵심 문단·불릿 → AI가 확장·정리**(처음부터 **풀 자동 작성 비권장**). 대화형으로 거시 구조를 잡은 뒤 스킬 호출할 것.  
5. **`munto-spec-review` 후**: 형식 통과≠제품 검증 → **기획/PM 사람 승인(Truth 레이어)** 필수 교육.  
6. (향후) 스킬에 **섹션별 금지 예시**·선행 입력 **강제 질문** 반영 검토 — 그 전까지 위 규격·리뷰로 방어(§3.5).  
7. **Spec 완료 게이트**(팀 정의·확정): SRS·원페이저에 더해 **`dev-chain-design` 으로 상위설계 산출물**(DBML, Swagger, Unit TCL 등)이 **합의된 상태**여야 하며, **특히 DBML·Swagger는 Spec에 포함되는 Peer Review 대상**이다. **개발자 간 충분한 Peer Review**를 마친 뒤에만 **`dev-chain-*` 구현·검증**(자동화 Chain 의 개발 단계)으로 넘긴다(AGENTS 의 Development Chain 과 동일 축).

### 4.4 SRS `1.2 Product Scope` — 표준 예시 함정과 올바른 스코프 (교육 메모)

**왜 문단을 추가했는지**  
`document/spec-standard.md` 에 붙어 있던 ✅ 예시가 **추천·점수 모델 등 특정 문제 영역**에 고정되어 있었다. 형식은 좋아 보여도, 교육·AI 작성 모두에서 「문장만 비슷하게 따라 쓴 스펙」이 나오기 쉽고, 오히려 **1.2가 해야 할 일(왜·무엇을·어디까지)** 을 가리키는 지침 역할을 약화시킨다.

**레포 조치**  
동 표준 문서의 1.2 가이드를 **스코프 절의 역할 정의**(기능 목록과 분리)·**도메인 비특정 예시**·**예시 복사 금지 안내**로 고쳤다.

**작성 원칙 요약** — 『소프트웨어 스펙의 모든 것』 등에서 말하는 **스코프** 관점과 맞춘다:

| 원칙 | 실무 해석 |
|------|-----------|
| **의사결정 문단** | 1.2는 7장의 앞머리가 아니라, **왜 지금 이 소프트웨어인지·무엇을 성공이라 부를지·책임 경계가 어디인지**를 한데 고정하는 자리다. |
| **문제·기회 → 목표 → 책임·경계** | 세 덩어리가 한 읽기 흐름으로 연결되어야 한다. **In/Out 불릿만** 두면 규격 미달에 가깝다. |
| **목표의 관측 가능성** | 가능하면 지표·관찰 결과와 연결한다. 반대로 **구현 알고리즘명까지 1.2에서 고정**하는 것은 피한다(합의는 요구·설계에서). |
| **예시는 빈칸 표시** | 템플릿 문장은 **베끼기용 소재가 아니라** 채워야 할 **항목**을 보여 주는 것이다. 도메인이 다른데 문장만 비슷하면 **형식 준수 가짜 스펙**이 된다. |

교육 시 **`spec-standard.md` 1.2 절**과 **`SRS_v3.3_template` 의 해당 헤더**를 함께 열어, 위 원칙이 템플릿 지침과 같은 방향인지 짚어 주면 좋다.

### 4.5 스펙·AI 협업 원칙(필수 메시지)

`munto-spec-writer` 는 **한 번에 채워지는 초안 SRS** 로 이어지기 쉽다. 스킬만 두고 보면 **스펙 품질보다 속도만 중시하는 구조**로 읽히기 때문에, 교육·리뷰에서는 아래를 **원칙**으로 고정하는 것을 권장한다.

| 원칙 | 설명 |
|------|------|
| **주도** | 작성자가 **범위·목적·거시 구조**를 깊게 고민하고 책임진다. |
| **AI는 보조** | 표준 준수, 문장·표 확장, **누락 가능 항목 질문** 등에 활용하고, AI 출력을 **검증 없이 수용하면 안 된다**. |
| **매몰·게으름 경계** | AI가 채워 준 문서에 고정되면 더 나은 대안을 검토하지 못하고, 바쁨과 맞물려 **형식만 채워진 SRS** 에 손 놓게 된다 → **품질·본인 역량** 모두 크게 깎인다. |
| **스킬 수정 전에도 교육으로 방어 가능** | §4.3의 **1.2→2.1→2.2 우선** 규격·§4.4 스코프 원칙·리뷰 게이트만으로도 위 허점을 상당히 줄일 수 있다(§3.5). |

### 4.6 SRS 템플릿(`SRS_v3.3`)이 있으면 섹션 의미는 해결되는가?

**충분히 가깝다 — 단, “있어서 해결되는 것”과 “항상 활용된다”는 다른 문제다.**

- `munto-spec-writer`는 `document/spec-standard.md` 및 **`document/spec-templates/SRS_v3.3_template.md`** 를 읽도록 되어 있다. 해당 템플릿에는 절별로 무엇을 써야 하는지 **한글·표준 SRS 유도 문**이 들어 있다(예: `2.6 Assumptions and Dependencies`는 **전제·선행 과제·통제 불가 외부요인·재사용 컴포넌트 종속**을 말하고, 「알려진 팩트 vs 가정」 구분까지 안내함).
- 따라서 **모델이 템플릿의 해당 절 지침까지 실제로 읽어 반영했다면**, “의미 자체를 모른다” 수준의 오류 비율은 **내려갈 수 있는 설계다**.

그럼에도 현장에서 틀리는 경우는 대략 아래처럼 남을 수 있다.

1. **로드 순서·범위**: 긴 SRS 작성 중 일부 장만 작성할 때, 템플릿의 해당 절 **전체**를 재확인하지 않고 학습 패턴으로 채운다.  
2. **언어 중첩**: “가정 깨지면 프로젝트 영향”“외부 불가항력”처럼 **운영 리스크 서술**이 2.6과 **내용이 비슷해** 보이나, 같은 말이라도 장의 **목적**(요구 근거 전제 vs 6장 품질·안전 속성 등)과 맞물려야 하는데 경계만으로는 헷갈릴 수 있다.  
3. **내용 채움 불가 정보**: 템플릿은 “무슨 카테고리를 적을지”는 알려 줄 뿠이, 인터뷰·경쟁 대응·사내 결정처럼 **비문서화 사실은 여전히 모른다**(컨텍스트 기아).

**실무 방어(교육·프롬프트)**: 작성 요청 시 `SRS_v3.3_template`의 **현재 작성 중인 절 헤더**를 명시해서 Read 하게 하거나, «2.6 직전까지 템플릿 해당 절 가이드를 인용했다»고 회신하게 요구하면 효과가 큼. **1.2 스코프**는 §4.4·`document/spec-standard.md` 와 함께 짚을 것. 스킬 본문에 “장 작성 직전 해당 템플릿 소절 Read”를 더 강하게 넣는 것은 **향후 보강 후보**.

### 4.7 기여 시 (하네스 수정 담당자)

- 수정은 **`.agents/` 우선**.  
- 새 래퍼 추가 후 **`check-adapters.sh` 로컬 실행** 습관화.
