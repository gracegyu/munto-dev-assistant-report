# munto-dev-assistant에 오케스트레이션 기반의 장시간 무인 에이전트 루프 체계 구현

| 항목 | 내용 |
| --- | --- |
| 배경 | Munto 개발팀은 사내 AI 툴 munto-dev-assistant로 에이전트 기반 개발을 하고 있다. 그러나 작업이 채팅 세션과 사용자 개입에 크게 묶여 있어, 동일 품질 기준을 유지하려면 사용자가 반복해서 지시하고 검증해야 한다. 해당 이슈를 통해 그 부담을 줄이고, 사용자 부재 상황에서도 맥락이 끊기지 않게 이어 가는 절차를 레포에 고정하고자 한다. |
| 현재 문제점 | • **작업 연속성**: 에이전트 작업이 채팅 세션 및 사용자 개입에 의존하여, 무인 상태에서 동일 맥락이 장시간 유지되기 어렵다.<br>• **완료 판정의 오류**: lint·test 등 검증 통과 여부가 확정되기 전에 작업이 종료되거나 "완료"로 처리되어, 대화상의 완료 선언과 레포의 실제 품질이 어긋날 수 있다.<br>• **표준화 부재**: 성공·실패·중단을 판단할 검증 명령·중단 조건·금지 작업이 팀 단일 규약으로 고정되어 있지 않아, 동일 요청이라도 처리 결과의 기준이 달라질 수 있다.<br>• **무인 실행 주체의 부재**: "장시간 무인으로 돌린다"는 요구에 맞는 반복 실행·검증·재시도를 실제로 수행하는 구성요소가 없거나 역할이 정해져 있지 않아, 문서·스킬만 있고 요구 동작이 재현되지 않을 수 있다. |
| 개선 방향 | 무인으로 요청된 작업을 끝까지 수행하기 위해, **오케스트레이션 구성요소**를 도입한다. 해당 구성요소는 **요청 식별 정보·작업 브랜치·검증 명령 목록·중단 조건(반복 횟수·시간·동일 오류 등)·개인 Slack 수신 설정**을 입력으로 받아, **코드 변경(또는 에이전트 호출) → 검증 실행 → 결과 판정**을 자동으로 반복한다.<br><br>완료(정상 종료)는 **합의된 검증 명령이 모두 성공**한 경우로만 정의하며, 그때까지는 **대화 종료나 부분 수정만으로 완료를 선언하지 않는다.** 검증 로그는 판정 근거로 보존한다. **검증 통과 판정은 에이전트의 자기 보고가 아니라, 오케스트레이터가 검증 명령의 exit code와 로그를 직접 파싱하여 결정한다.**<br><br>**정상 종료가 확정된 직후**, 사전에 등록한 **개인 Slack**으로 요청 요약·통과한 검증·브랜치·커밋·PR 링크(해당 시)를 전송한다. **비정상 종료** 시 Slack으로 종료 원인을 통지한다.<br><br>본 오케스트레이터는 외부 워크플로우 프레임워크 도입 없이, **Claude Code를 에이전트 런타임으로 그대로 활용**하는 얇은 구성으로 설계한다. 루프 형태는 "에이전트 호출 → 검증 명령 실행 → 결과 판정 → 재시도/종료"의 단순 반복으로 한정한다(복잡한 다단계 토폴로지가 필요해지는 시점에 재검토).<br><br>**실행 주체: GitHub Actions** 를 채택한다. `workflow_dispatch` 또는 이슈/코멘트 트리거로 작업을 개시하며, 매 실행마다 격리된 러너에서 수행한다. 사용자 머신 가동 여부와 무관하게 동작하며, 시간·CPU 한도와 감사 로그·시크릿 관리가 기본 제공된다. 로컬 데몬/사내 VM은 "사용자 머신 의존" 또는 "운영 부담"으로 무인 요구사항과 충돌하므로 제외한다. |
| 고려요소 | • **비용·한도:** 토큰·API·러너 CPU·시간에 **상한**을 두고, 상한 도달 시 **비정상 종료**로 처리한다. 무한 재시도는 **비용·안전** 측면에서 허용하지 않는다.<br>• **보안·권한:** 프로덕션 DB·시크릿·무승인 배포 등 **고위험 작업은 기본 금지 사항**으로 설정한다.<br>• **알림:** 작업 종료 시 사전에 입력한 개인 Slack을 통해 정상 종료/비정상 종료 여부를 발송한다. |
| 기대효과 | • **무인 연속 작업:** 사용자 개입 없이 **동일 작업 맥락**이 유지되어, 요청 단위 작업을 **끝까지 수행**할 수 있는 기반이 마련된다.<br>• **가시성:** 작업 종료 시 **개인 Slack**으로 결과가 전달되어, **요청자가** 완료 여부와 근거를 확인할 수 있다. |

## comment by 전규현

현재 munto-dev-assistant 전반을 분석해 개선 방향(TO-BE)을 정리하는 중인데, 본 이슈의 기본 방향(검증 주체 분리·GitHub Actions 채택·얇은 루프)은 적절하다고 봅니다. 다만 무인 루프가 사람 리뷰 게이트(스펙·설계 베이스라인, PR 머지)까지 자동 통과시키면 *통과 = 인수* 원칙이 무너질 위험이 있어, 무인화 범위는 *PHASE 2 구현·자동 검증·PR 생성까지*로 한정하는 것이 안전합니다. 구현은 그대로 진행하되, 곧 TO-BE 를 공유해 드릴 테니 적용 시 함께 맞춰 가면 좋겠습니다.
