## SRS (Software Requirements Specification)

### 항목 구조

아래는 SRS Template v3.3의 **고정 항목**입니다. 항목의 추가, 삭제, 번호 변경은 불가합니다.

```
1. Introduction (개요)
   1.1 Purpose (목표)
   1.2 Product Scope (범위)
   1.3 Document Conventions (문서규칙)
   1.4 Terms and Abbreviations (정의 및 약어)
   1.5 Related Documents (관련문서)
   1.6 Intended Audience and Reading Suggestions (대상 및 읽는 방법)
   1.7 Project Output (프로젝트 산출물)
       1.7.1 Output Format (산출물 형태)
       1.7.2 Output Name and Version (산출물명(가칭) 및 버전)
       1.7.3 Patent Information (특허 출원 유무 및 내용)

2. Overall Description (전체 설명)
   2.1 Product Perspective (제품 조망)
   2.2 Overall System Configuration (전체 시스템 구성)
   2.3 Overall Operation (전체 동작방식)
   2.4 Product Functions (제품 주요 기능)
   2.5 User Classes and Characteristics (사용자 계층과 특징)
   2.6 Assumptions and Dependencies (가정과 종속 관계)
   2.7 Apportioning of Requirements (단계별 요구사항)
   2.8 Backward compatibility (하위 호환성)

3. Environment (환경)
   3.1 Operating Environment (운영 환경)
       3.1.1 Hardware Environment (하드웨어 환경)
       3.1.2 Software Environment (소프트웨어 환경)
   3.2 Product Installation and Configuration (제품 설치 및 설정)
   3.3 Distribution Environment (배포 환경)
       3.3.1 Master Configuration (마스터 구성)
       3.3.2 Distribution Method (배포 방법)
       3.3.3 Patch/Update Method (패치와 업데이트 방법)
   3.4 Development Environment (개발 환경)
       3.4.1 Hardware Environment (하드웨어 환경)
       3.4.2 Software Environment (소프트웨어 환경)
   3.5 Test Environment (테스트 환경)
       3.5.1 Hardware Environment (하드웨어 환경)
       3.5.2 Software Environment (소프트웨어 환경)
   3.6 Configuration Management (형상관리)
       3.6.1 Location of Outputs (산출물 위치)
       3.6.2 Build Environment (빌드 환경)
   3.7 Bugtrack System (버그트래킹)
   3.8 Other Environment (기타 환경)

4. External Interface Requirements (외부 인터페이스 요구사항)
   4.1 System Interfaces (시스템 인터페이스)
   4.2 User Interface (사용자 인터페이스)
   4.3 Hardware Interface (하드웨어 인터페이스)
   4.4 Software Interface (소프트웨어 인터페이스)
   4.5 Communication Interface (통신 인터페이스)
   4.6 Other Interface (기타 인터페이스)

5. Performance requirements (성능 요구사항)
   5.1 Throughput (작업처리량)
   5.2 Concurrent Session (동시 세션)
   5.3 Response Time (대응시간)
   5.4 Performance Dependency (성능 종속 관계)
   5.5 Other Performance Requirements (기타 성능 요구사항)

6. Non-Functional Requirements (기능 이외의 요구사항)
   6.1 Safety requirements (안전성 요구사항)
   6.2 Security Requirements (보안 요구사항)
   6.3 Software System Attributes (소프트웨어 시스템 특성)
       6.3.1 Availability (가용성)
       6.3.2 Maintainability (유지보수성)
       6.3.3 Portability (이식성)
       6.3.4 Reliability (신뢰성)
       6.3.5 Remaining Attributes (나머지 특성)
   6.4 Logical Database Requirements (데이터베이스 요구사항)
   6.5 Business Rules (비즈니스 규칙)
   6.6 Design and Implementation Constraints (설계와 구현 제한사항)
       6.6.1 Standards Compliance (표준준수)
       6.6.2 Other Constraints (기타 제한 사항)
   6.7 Memory Constraints (메모리 제한 사항)
   6.8 Operations (운영 요구사항)
   6.9 Site Adaptation Requirements (사이트 적용 요구사항)
   6.10 Internationalization Requirements (다국어 지원 요구사항)
   6.11 Unicode Support (유니코드 지원)
   6.12 64bit Support (64비트 지원)
   6.13 Certification (제품 인증)
   6.14 Field Test (필드 테스트)
   6.15 Other Requirements (기타 요구 사항)

7. Functional Requirements (기능요구사항)
   7.1 대분류 기능1
       7.1.1 ...
       7.1.1.1 ...
   7.2 대분류 기능2
   7.3 대분류 기능3
   ...
```

### 섹션별 작성 지침

#### 1장 Introduction

| 섹션 | 작성 포인트 |
| --- | --- |
| **1.1 Purpose** | **이 _문서_ 가 왜 작성됐는지** (제품의 목적이 아님). 제품의 revision/release 번호 명시. 작성 가이드는 아래 "1.1 작성 가이드" 참조 |
| **1.2 Product Scope** | 아래 "1.2 작성 가이드" 참조 |
| **1.3 Document Conventions** | 우선순위(P1/P2/P3) 정의, 하위 요구사항 우선순위 상속 여부 명시. **수식/기호가 예시로 등장하는 경우 해당 위치에서 간략 설명하거나 1.4 참조를 명시** |
| **1.4 Terms and Abbreviations** | **본 문서의 이해관계자 기준, 일부가 모를 수 있는 용어·약어만** 정의한다. **모든 용어를 다 적지 않는다.** 모두가 뻔히 아는 용어(예: `JWT`, `TCP/IP`)는 적지 않고, 특정 챕터만 보는 사람용의 전문 용어(예: 개발자만 보는 4 장의 알고리즘 용어)도 여기에 정의할 필요 없다. 작성 규칙·판정 기준은 공통 작성 규칙 §"용어·약어(1.4) 작성 원칙" 참조 |
| **1.5 Related Documents** | Notion 정책문서, Figma, GitHub(API/ERD), Jira 이슈 링크 포함. **시간이 흘러도 끊어지지 않는 링크**(베이스라인이 설정된 안정 버전)를 우선 사용. 작성 가이드는 아래 "1.5 작성 가이드" 참조 |
| **1.6 Intended Audience** | 이해관계자별로 읽어야 하는 챕터·읽는 방법 가이드. **챕터 × 역할 매트릭스**(아래 "1.6 작성 가이드" 참조) 형태가 표준 |
| **1.7.1 Output Format** | 제품/라이브러리/툴 구분, 플랫폼(앱/백엔드/SDK) 명시. **흔한 실수**: 프로젝트 산출 _문서_ 목록을 적는 것 — 여기는 *최종 SW 산출물의 형태*를 적는 곳 |
| **1.7.2 Output Name and Version** | 산출물명(가칭 OK), 레포지토리명, 초기 버전 명시 |
| **1.7.3 Patent Information** | 특허 출원 가능 아이디어 여부. 해당 없으면 `None` |

**1.1 Purpose 작성 가이드:**

이 챕터는 _문서의 목적_ 을 적는 곳입니다. _제품의 목적_ 은 1.2 에서 적습니다. 둘을 혼동하면 같은 내용이 문서 여러 곳에 중복됩니다.

문서의 목적은 _이 SRS 가 어떤 맥락에서 누구를 위해 작성되었는지_ 를 가리킵니다. 예:

- _"본 SRS 는 내부 개발용으로 작성되었으며, 외부 외주 공유 시 §6.6 등 회사 내부 사정 관련 내용은 제거 후 전달한다."_
- _"v2.0 개발용. v1.0 경험이 있는 노련한 개발자를 투입할 예정이라, 분석·설계 시간을 단축하기 위해 일반적인 신규 프로젝트 SRS 보다 간결하게 작성한다. 본 문서를 일반 SRS 작성 참고용으로 보지 말 것."_
- _"신입 개발자 스펙 작성 교육 교재로도 사용. 따라서 일반 프로젝트보다 상세한 근거·예시를 포함."_
- _"공동 개발 파트너사(KAIST)와 공유 — 외부 공유에 적합한 수준으로 작성."_

**흔한 실수 (절대 하지 말 것)**

- ❌ *제품의 목적*을 적고, 제품 기능을 잔뜩 나열 — 이건 1.2 와 7 장의 영역이며, 중복으로 인한 유지보수 비용 폭발
- ❌ "본 문서는 OOO 제품의 SRS 다" 한 줄만 적기 — "소설 앞에 _본 책은 소설이다_ 적은 것" 과 같음 (틀린 말은 아니지만 의미가 없음)
- ❌ 대상 독자 목록만 적기 ("이 문서의 대상자는 개발, 마케팅, 영업, QA") — 대상 독자는 1.6 의 영역

---

**1.2 Product Scope 작성 가이드:**

> **SRS 전체에서 가장 중요한 챕터입니다.** 거의 모든 이해관계자(경영진, PM, 기획, 디자이너, 개발자, QA, 마케팅, 영업)가 _유일하게 빠짐없이 읽는_ 챕터이며, 경영진은 이 챕터만 보고 _프로젝트 GO/NO-GO_ 를 판단할 수 있어야 합니다. AI 는 1.2 를 다른 어떤 챕터보다 _신중하게_ 작성해야 합니다.

경영진 요약(executive-level summary) 수준으로 **"왜 이 소프트웨어가 필요한지"**를 중심으로 작성합니다.

**작성 원칙 (4가지):**

1. **비즈니스 배경 (1문단)**: 현재 방식의 문제점과 이번 소프트웨어가 해결하려는 비즈니스 목표·기대 효과를 서술합니다. 기획 문서나 Jira에 이미 배경·기대효과가 정리되어 있으면 핵심만 요약하고 해당 문서를 참조 링크로 첨부합니다.
2. **소프트웨어의 핵심 역할과 경계 (1문단)**: 이 소프트웨어가 하는 것과 하지 않는 것을 한두 문단으로 서술합니다. In/Out of Scope를 나열식으로 쓰지 않습니다.
3. **In Scope 항목 전체 나열 금지**: 개별 요구사항은 이후 섹션(2장, 7장)에서 상세히 다루므로, 여기서는 전체 목록을 열거하지 않습니다. 대표 기능 2~3개만 한 문장 안에 녹여 언급합니다.
4. **Will not do (지원하지 않을 것) 명시 — 필수**: _충분히 지원할 법한데 의도적으로 빼는 핵심 기능_ 만 콕 집어서 명시합니다. 미지원 기능을 모두 나열할 필요는 없습니다. _왜_ 안 하는지 한 줄 근거를 함께 적습니다.
   - 예: _"본 SRS 는 디지털 카메라만 지원하며, 아날로그 카메라는 v2.0 에서 별도 SRS 로 다룬다."_
   - Will not do 누락은 후속 단계에서 _이해관계자가 지원하는 줄로 오해_ → 분쟁의 핵심 원인입니다.

**Why–What–How 균형 원칙 (모든 섹션 공통, 1.2 에 특히 강하게 적용):**

| 비중          | 위치                         | 내용                                                                 |
| ------------- | ---------------------------- | -------------------------------------------------------------------- |
| **Why ~30%**  | 1.2, 2.1, 7.x 각 기능 도입부 | 왜 만드는가 — 비즈니스 목표·문제·기대 효과                           |
| **What ~50%** | 2장, 7장 본문                | 무엇을 만드는가 — 외부에서 본 동작·입출력                            |
| **How ~20%**  | 4장, 6.6, 7.x 구현 노트      | 어떻게 만드는가 — 핵심 제약·결정 사항만 (상세 구현은 설계 문서 영역) |

- _정확함 ≠ 자세함_: 1.2 는 분량이 적어도 Why 가 누락되면 불완전합니다. 반대로 What/How 가 5 페이지여도 Why 가 1 줄이면 1.2 의 목적을 달성하지 못한 것입니다.
- 1.2 에서 핵심 아키텍처 방향(예: _"베이지안 점수 체계로 전환"_, _"MSA 로 분리"_, _"결제 게이트웨이를 외부 PG 로 위탁"_, _"채팅을 Socket.IO 로 통일"_)을 한 문장이라도 언급하면, **공통 작성 규칙 §"핵심 결정 — 대안 검토 박스"** 에 따라 별도 박스를 추가합니다.

**기술 스펙 나열에 빠지지 말 것 (안티패턴):**

> _"엔진출력 200마력, 차 길이 4.5m, 타이어 사이즈 215"_ — 이것만 보면 _어떤 자동차인지_ 모릅니다. _"기존 자동차보다 연비를 획기적으로 개선한 그린 컨셉 자동차로, 3년 내 친환경 차 시장을 석권한다"_ — 이렇게 적어야 _왜·무엇을_ 만드는지 한 눈에 들어옵니다.

소프트웨어도 동일합니다. _"베이지안 점수 산출 / Redis 캐시 / 큐 길이 100"_ 식 나열이 아니라, _어떤 사업 가치를 만들기 위한 어떤 종류의 소프트웨어인가_ 를 한 문단으로 드러내야 합니다.

**분량 가이드 (강제 제약):**

- _1~3 분 안에 읽을 수 있는 분량_ — 보통 반 페이지 ~ 한 페이지.
- 너무 길면 핵심이 묻혀, 이해관계자마다 _다른 부분을 기억_ 하게 됩니다. 정렬(alignment) 효과 상실이 1.2 의 가장 큰 실패 모드입니다.
- 분량보다 _Why 가 한 문단 안에 명확히 드러나는가_ 가 우선입니다.

**흔한 실수 (절대 하지 말 것):**

- ❌ 상세한 기능을 장황하게 잔뜩 나열 — 자동차 명세서를 적는 것과 같음
- ❌ 마케팅용 슬로건·이미지를 잔뜩 포함 — 1.2 는 _경영진·실무자의 의사결정 도구_ 이지 광고가 아님
- ❌ _Will not do_ 누락 — 후속 단계에서 이해관계자가 지원하는 줄로 오해
- ❌ Why 없이 What 만 적기 (_"이 시스템은 베이지안 점수를 산출한다"_ 만 적고 _왜_ 가 빠짐)

**작성 주체 — AI 가 가장 조심해야 할 부분:**

- 1.2 는 _제품 기획자·경영진의 의도_ 를 담는 곳입니다. **개발자나 AI 가 추측해서 쓰면 안 됩니다.**
- 입력 자료(기획 문서·Jira·1:1 의사결정 기록)에 _비즈니스 목표가 명확히 적혀 있지 않으면_, AI 는 **본문을 추측해서 채우지 말고** 사용자에게 _"이 프로젝트의 비즈니스 목표·기대 효과를 한 문단으로 알려 주세요"_ 라고 먼저 물어야 합니다.

**다중 SRS 일관성 (조건부 — 대형 프로젝트에만 해당):**

- 상위/하위 SRS 가 분리된 경우, 하위 SRS 의 1.2 는 상위 SRS 의 1.2 와 _모순되지 않아야_ 합니다.
- 같은 내용을 여러 SRS 에 _복사 금지_. 한 곳에만 적고 나머지는 _링크_ 로 참조합니다 (1.5 Related Documents 가이드 참조).

```markdown
## 1.2 Product Scope

기존 가중합 방식은 행동 지표 간 변별력이 낮아 좋아요 전환율과의 상관관계가 약하다는 문제가 있다. 본 프로젝트는 베이지안 기반 점수 체계로 전환하여, 추천 정확도를 높이고 좋아요 전환율을 개선하는 것을 목표로 한다. (상세 배경: [Jira PROJ-000] / [기획 문서])

- 결제·구독 기능 — 기존 결제 시스템을 그대로 사용하며, 본 SRS 의 범위가 아니다.
```

상위 기능의 우선순위가 하위 기능의 우선순위보다 _낮을 수는 없습니다_ (낮다는 것은 P2 < P1, 즉 숫자가 클수록 우선순위 낮음).

- ✅ 상위 §7.1 (P1) 인데 하위 §7.1.1 (P2) — *전체 중 일부는 다음 버전으로 미룬다*는 의미 OK

**버전·커스터마이징 표기 양식:**

```markdown
- 회원가입 기능 ← 현재 베이스라인 (표기 없음)

- 화이트라벨 로고 (L사) ← L 사 전용
- 채팅 비활성화 (only H사) ← H 사에만 적용
```

- 커스터마이징이 _너무 많으면_ 본 SRS 에 섞지 말고 **별도 문서로 분리**합니다 (공통 SRS + 고객사별 SRS).
- 후속 버전 표기는 _추가된 것_ 만 표시하면 충분합니다 — 베이스라인(v1.0) 내용에는 표기를 붙이지 않습니다.

**색깔만으로 정보 전달 금지 (책 §1.3 명시 — 현대화 확장):**

- ❌ "변경된 내용은 _빨간 글씨로_ 표시" — 인쇄·컬러블라인드·스크린리더에서 정보 손실
- ✅ "변경된 내용은 _(변경: 2026-05-12)_ 와 같이 _텍스트로_ 표기하고, 색깔은 _보조_ 표시만"

**문서 변경 추적 운영 (Notion / GitHub 기준):**

- 공동 작업 중에는 Notion 의 _댓글·페이지 히스토리_ 또는 GitHub PR 의 _diff_ 로 변경분이 보이게 합니다.

**2.1 Product Perspective 작성 가이드:**

**목적**: 제품을 _외부에서 본 모습_. 우리가 만들 시스템과 _외부 시스템들 간의 관계도_.

**핵심 가치 — 외부 시스템 누락 방지:**

- 출시 후 누락된 외부 인터페이스가 발견되면 *몇 배의 노력*으로 재작업하게 됩니다.
- 본 챕터는 *외부 시스템을 빠짐없이 식별*하기 위해 존재합니다.
- AI 는 _모든 이해관계자가 본 다이어그램을 함께 보며 누락 검토할 수 있도록_ 작성합니다.

**작성 원칙:**

1. **컨텍스트 다이어그램 필수** — 우리 시스템을 중앙에 두고, 모든 외부 시스템·외부 사용자(Actor)를 연결한 다이어그램. Mermaid 또는 이미지.
2. **외부 시스템 목록 명시** — 다이어그램 아래에 각 외부 시스템의 _역할_ 을 한 줄씩 기술.
   - 예: _"Game center — 애플사의 게임센터로서 게임 스코어와 미션을 관리하는 서버"_
3. **인터페이스 _유무_ 만 표시** — 여기서는 인터페이스의 _상세 프로토콜·메서드·payload_ 는 적지 않습니다. _연결되어 있다_ 는 사실만.
   - 상세 인터페이스는 §4 (External Interface Requirements) 의 영역입니다.

**식별해야 할 외부 시스템 카테고리 (자체 점검 체크리스트):**

| 카테고리              | 예시                                                           |
| --------------------- | -------------------------------------------------------------- |
| 외부 사용자 (Actor)   | User, Admin, Guest, 운영자, 파트너사 직원                      |
| 외부 서비스 (B2B)     | 결제 PG, SMS/이메일, 광고 서버, 분석 서버, 푸시 게이트웨이     |
| 외부 플랫폼           | App Store, Google Play, Apple Game Center, OAuth(Google/Kakao) |
| 외부 데이터 소스      | Daily puzzle server, 환율 API, 지도 API                        |
| 회사 내부 다른 시스템 | 사내 인증, 정산 시스템, CRM, 데이터 웨어하우스                 |

AI 는 _위 5개 카테고리를 각각 점검_ 하여 누락이 없는지 확인합니다.

**2.2 Overall System Configuration 작성 가이드:**

**목적**: 제품의 _내부_ 모습. 주요 컴포넌트 + 그들 간의 연결 관계.

**작성 원칙:**

1. **상위 설계의 시작** — 너무 깊숙이 들여다보지 않습니다. _주요 컴포넌트 3~7 개_ 수준이 적당합니다. 10 개 넘으면 1 단계 더 추상화합니다.
2. **컴포넌트 도출 기준은 다양** — 프로젝트 성격·환경·개발자 구성에 따라 다릅니다. 흔한 기준:
   - _기능의 독립성_ — 서로 다른 책임 영역
   - _배포 단위_ — 마이크로서비스 분리
   - _기술 스택_ — 모바일 앱 vs 백엔드 vs 어드민 웹
3. **AI 는 도출 기준을 _명시_ 해야 함** — 같은 컴포넌트 분할이라도 _왜 그렇게 나눴는지_ 한 문장 근거를 첨부합니다.

**1.2 ↔ 2.2 일관성 룰:**

1.2 에서 _"MSA 로 분리"_, _"결제 게이트웨이 외부 PG 위탁"_ 같은 핵심 아키텍처 방향을 언급했다면, 2.2 의 컴포넌트 도식이 그 결정을 _그대로 반영_ 해야 합니다. 1.2 ↔ 2.2 가 모순되면 큰 문제입니다. AI 는 1.2 의 핵심 결정 박스(공통 작성 규칙 §"핵심 결정 — 대안 검토 박스")를 먼저 확인한 뒤 2.2 를 작성합니다.

**작성 예 (모바일 스도쿠 게임 앱):**

- Game app 내부를 _Main controller / Info manager / Puzzle generator_ 3 개로 분리
- 분할 근거: _기능의 독립성에 따라_
- 각 컴포넌트의 _외부 시스템과의 연결_ 도 함께 표시 (2.1 의 외부 시스템이 어느 내부 컴포넌트로 연결되는지)

**스펙 vs 설계 — 어디까지 스펙에 담는가:**

_"스펙은 What, 설계는 How"_ 라는 흔한 구분은 _반만 맞습니다_. 잘 분석된 스펙은 _상당 부분 설계 영역까지_ 다룹니다. 설계를 너무 깊이 적으면 구현 개발자의 자유도를 빼앗고, 너무 얕게 적으면 구현 단계에서 핵심 결정을 떠넘기게 됩니다. AI 는 _명시적 기준_ 으로 결정해야 합니다.

**Munto 의 기본 경계 (내부 개발 + 중규모 + dev-chain 사용):**

| 항목 | 위치 | 이유 |
| --- | --- | --- |
| 외부 인터페이스 (REST/Webhook 시그니처·payload·에러) | **스펙** (§4 · §7) | 호출하는 쪽 입장에서는 _반드시 지켜야 하는 계약_. 자유도 없음 |
| DB 논리 스키마 (엔티티·관계·핵심 제약) | **스펙** (§6.4 + dev-chain-design 의 DBML) | 도메인 모델은 _기능과 분리 불가_ |
| 알고리즘/라이브러리 제약 (라이선스·확장성 사유) | **스펙** (§6.6 Design and Implementation Constraints) | 제약 자체가 _요구사항_ |
| DB 물리 설계 (인덱스·파티셔닝·튜닝) | **설계** (dev-chain-design / 백엔드 결정) | 성능 요구사항을 만족하기 위한 _수단 선택_ |
| 내부 클래스·모듈 분리, 내부 알고리즘 선택 | **설계** (dev-chain-{backend,frontend,mobile}) | 외부 행위가 같으면 _교체 가능_ |
| 컴포넌트 분리 (2.2 수준 — 마이크로서비스, 앱/웹/백엔드 등) | **스펙** (§2.2) | 인력 배치·기술 스택·외주 분할에 영향 |
| 컴포넌트 _내부_ 의 패키지·레이어 구조 | **설계** | 같은 외부 행위 보장하면 무관 |

**판정 규칙 (AI 가 적용할 결정 트리):**

1. _이 결정이 바뀌면 외부에서 보이는 동작·계약·일정·예산이 바뀌는가?_ → **Yes 면 스펙**.
2. _이 결정이 다른 컴포넌트나 다른 개발자에게 제약을 주는가?_ → **Yes 면 스펙**.
3. *이 결정이 하나의 컴포넌트 내부에서 *교체 가능* 한가?* → **Yes 면 설계**.
4. _법·라이선스·표준·인증 요구로 결정된 것인가?_ → **무조건 스펙** (§6.6).

**원칙 정리 (책 §2.8 인용):**

- 스펙과 설계는 _칼로 자르듯 구분되지 않습니다_.
- 스펙에는 _크고 작은 설계 내용이 포함_ 됩니다.
- _스펙 작성 후 별도의 설계 문서가 반드시 필요한 것은 아닙니다_. Munto 는 _스펙 + dev-chain-design 산출물(DBML/Swagger/TCL)_ 조합으로 설계를 대체합니다.
- 다만 _스펙 문서는 필수_, _설계 문서는 선택_ — 이 비대칭은 절대 뒤집히지 않습니다.

**AI 안티 패턴 (스펙·설계 경계 오작성):**

- ❌ 2.2 에 _클래스 다이어그램_ 까지 그림 — 설계 영역으로 넘어감
- ❌ 7 장 기능에 _"Redis 를 사용한다"_ 만 적고 _왜_ 를 누락 — 제약이면 §6.6 으로, 단순 구현 선택이면 _스펙에서 빼고_ dev-chain-design 으로
- ❌ DB 논리 스키마(엔티티·관계)를 _스펙 어디에도 안 적고_ dev-chain-design 의 DBML 에서 처음 등장 — 도메인 모델은 *스펙*에 있어야 함
- ❌ 외부 인터페이스를 _"개발 시 결정"_ 으로 미룸 — 외부 인터페이스는 _반드시 스펙_

**2.3 Overall Operation 작성 가이드:**

**목적**: 2.2 의 컴포넌트 구성을 기준으로 _주요 시나리오_ 가 어떻게 동작하는지 설명.

**작성 원칙:**

1. **주요 시나리오만 — 모든 시나리오 X** — 전체 시스템 구성에 표시된 컴포넌트들이 _대체로 어떻게 동작하는지_ 알 수 있는 _핵심 4~6 개_ 정도.
   - 7 장에서 상세 기능별로 작성하므로 여기서는 _대표 흐름_ 만.
2. **서술식 vs 시퀀스 다이어그램** — 둘 다 OK. 단:
   - **2 장은 일반인도 읽는 챕터** → _서술식 권장_.
   - 서술식 장점: _적절한 설명 덧붙이기 쉬움, 수정 용이, 툴 불필요._
   - 시퀀스 다이어그램은 _컴포넌트 간 메시지 흐름이 핵심_ 인 경우에만 (예: 결제 콜백, OAuth, 비동기 분산 처리).
3. **서술식 작성 양식**:
   - _컴포넌트 이름을 주어·목적어_ 로 사용 (모호한 자연어 _"시스템이 알아서 처리한다"_ 금지).
   - 시나리오별 5~10 단계 정도.
   - 예: _"Main controller 는 Ads server 에 광고 생성을 요청하여 화면 하단에 배너 광고를 Display 한다"_

**시나리오 선정 기준 (어떤 시나리오를 골라야 하나):**

- 가장 자주 실행되는 _메인 흐름_ (해피 패스)
- _외부 시스템 다수가 연동_ 되는 흐름 (외부 연결 검증)
- _다른 시나리오와 패턴이 크게 다른_ 보조 흐름 (예: 게임 수행 vs 설정 변경)
- 보안·결제 등 _위험도가 높은_ 흐름

**2.4 Product Functions 작성 가이드:**

**목적**: 7 장에서 상세히 다룰 _주요 기능_ 들을 한눈에 볼 수 있게 요약.

**작성 원칙:**

1. **7 장 섹션 제목과 1:1 매핑 필수** — 2.4 에 등장하는 기능명이 7 장 §7.x 의 섹션 제목으로 그대로 옮겨져야 합니다. 양쪽이 어긋나면 추적 불가.
2. **목록 형태 (서술식 X)** — 읽는 사람이 기능을 _쉽게 스캔_ 할 수 있게.
3. **고객의 언어로 작성** — 전문 용어를 써야 할 때도 _이 SRS 를 읽을 고객의 계층_ 에 맞춰서:
   - _카메라 SRS_ → 카메라를 살 사람이 알 수 있는 카메라 전문 용어 OK
   - _의료 시스템 SRS_ → 의사가 아는 의학 용어 OK
   - _일반 소비자 앱_ → 일반인 용어로
4. **잘 작성된 브로셔처럼** — 내용을 읽으면 _제품의 특성이 이해되고, 매력적으로 보여야_ 합니다.

**금지 (흔한 실수):**

- ❌ _"사용자 추가", "사용자 삭제"_ 같이 _큰 의미 없는 작은 기능_ 을 나열
- ❌ 단어식으로만 나열 (_"로그인 / 회원가입 / 결제"_ 같이 한 단어)
- ❌ 다이어그램에 _설계 내용_ 을 잔뜩 그려넣기 — 2.4 는 기능 설명용, 설계 영역 아님
- ❌ 7 장과 어긋나는 기능명 사용 — 추적 불가능해짐

**좋은 작성:**

- ✅ 제품을 _대표할 수 있는 기능_ 을 적당한 설명과 함께
- ✅ _왜 매력적인지_ 한 줄 설명 첨부 (브로셔 톤)
- ✅ 그림·사진은 _기능 설명에 도움이 되는 것만_ 첨부 (장식용 X)

**2.5 User Classes 작성 가이드:**

**목적**: 본 SW 를 사용할 _모든_ 계층을 식별합니다. 최종 사용자만이 아니라 *운영자·관리자·기술지원·B2B 파트너*까지 포함해야 누락이 없습니다.

**식별 대상 카테고리 (체크리스트 — 4 종 모두 점검):**

| 카테고리           | 예시                                        |
| ------------------ | ------------------------------------------- |
| 최종 사용자        | 일반 사용자 (Free / Premium), 게스트        |
| 운영자·관리자      | 백오피스 어드민, 컨텐츠 큐레이터, CS 운영자 |
| 기술 지원·인프라   | DevOps, SRE, DBA                            |
| 파트너·외부 협력자 | B2B 파트너, API 외부 클라이언트             |

**각 계층별 작성 항목 (5 종):**

1. _사용 빈도_ — 일 1 회 / 주 1 회 / 월 1 회 / 이벤트 시
2. _주로 사용하는 기능_ — §7.x 의 어느 기능군
3. _보유 기술·전문 지식·교육 수준_
4. _권한 수준_ — 읽기 / 쓰기 / 관리자 / 시스템
5. _중요도_ — 본 SW 의 핵심 사용자인가 (단순 사용자 수만이 기준이 아님)

**광범위 사용자 추가 분류 룰 (책 §2.5 명시):**

사용자가 매우 광범위 (오피스 SW, 범용 앱 등) 해서 한 계층으로 묶기 어려우면, 같은 "최종 사용자" 안에서도 _나이·경험·직업·숙련도_ 로 분할합니다.

- 예: 웹 프레젠테이션 SW — _초·중·고등학생 / 초급자_ vs _직장인 / 숙련자_ 분리 (사용 기간·필요 기능·협업 빈도 모두 다름)
- 예: 유아용 앱 — _부모(주 구매자, 옵션 설정)_ vs _아기(주 사용자, 터치 조작)_ 분리

**잘못된 예:**

- ❌ "사용자" 한 줄로 끝 — 운영자·관리자·기술지원 누락
- ❌ 사용 빈도·권한·중요도 없이 _이름만_ 나열
- ❌ "모든 사람" 같이 식별이 아무 의미 없는 범주

**2.6 Assumptions and Dependencies 작성 가이드:**

**목적**: 본 프로젝트가 _통제할 수 없는 외부 요소_ 또는 *아직 확정되지 않은 전제*를 명시합니다. 리스크 관리의 시작입니다.

**작성 대상 (4 카테고리):**

| 카테고리                                | 예시                                                      |
| --------------------------------------- | --------------------------------------------------------- |
| 미확정 기술의 _성공 가정_               | "ML 엔진은 알파 이전까지 개발 성공한다고 가정"            |
| 외부 라이브러리·SDK 의 _정식 출시 가정_ | "M 사 A 라이브러리 정식 버전이 베타 이전 출시"            |
| _외부 의사결정_ 결과 가정               | "11 월 국회 통과 예정인 세무 법률은 원안대로 통과"        |
| 외부 _수령 마감_                        | "Audio device driver 5/15 까지 제공받아야 일정 차질 없음" |

**잘못된 예 (책 §2.6 명시):**

- ❌ "버그를 완전히 해결하지 못하면 QA 팀과 협의하여 출시 결정" — _별 의미 없는 일반론_. 운영 전략은 §1.2 / §8 (변경 관리) 의 영역
- ❌ 통제 *불가능*하지도 않고 _프로젝트 일정·아키텍처에 실제 영향_ 도 없는 일반 사실을 다 나열

**각 가정에 _영향 받는 섹션_ 도 함께 표기 (TBD 4 항목 룰과 동일):**

```markdown
- ML 엔진 알파 이전 개발 성공 가정 (실패 시 영향: §7.2 추천 알고리즘은 룰베이스로 대체, §5.3 응답시간 목표 1 s → 3 s 완화)
```

**2.7 Apportioning of Requirements 작성 가이드:**

**목적**: 기능을 _현재 버전(v1.0)_ 과 _미래 버전(v2.0, v3.0...)_ 으로 분배합니다. 첫 버전은 *기본에 충실*하고, *미래 계획은 *현재 아키텍처 결정에 미리 영향\*을 줍니다.

**핵심 룰 — 미래 계획이 _지금_ 아키텍처에 영향을 주는 경우 (책 §2.7 명시):**

| 미래 계획                                 | 현재 아키텍처에 미치는 영향                                                  |
| ----------------------------------------- | ---------------------------------------------------------------------------- |
| v2.0 에 해외 시장 진출                    | 지금 §6.10 국제화(i18n) 카테고리를 결정 (LC_MESSAGES·LC_TIME·LC_MONETARY 등) |
| v2.0 에 iOS·Windows 지원                  | 지금부터 _플랫폼 의존 코드 분리_ (UI 추상화 / 로컬 스토리지 추상화)          |
| 1 년 안에 100 사용자 → 100 만 사용자 확장 | 지금부터 _확장 가능한 아키텍처_ (DB 샤딩·캐시·CDN·메시지 큐)                 |

**2.8 Backward Compatibility 작성 가이드:**

**목적**: 기존 시스템을 *업그레이드*하는 경우, 기존 데이터·기존 클라이언트·기존 외부 연동과의 호환 정책을 정의합니다.

**원칙:**

- 최초 개발이라면 `N/A` 한 줄로 충분.
- 업그레이드인 경우 *호환 vs 포기*를 *모두 명시*합니다. 호환을 *포기*하는 것도 결정입니다 (모든 것을 호환하면 아키텍처가 과거에 묶여 발전 불가).

**호환 결정 시 정의할 4 가지:**

1. _호환 대상_ — 어떤 기존 데이터·파일·API 버전을 호환하는가
2. _호환 방식_ — 자동 마이그레이션 / 호환 모드 동작 / 별도 변환 도구 제공
3. _호환 포기 대상_ — 어떤 것은 _호환하지 않기로_ 했는가 (이것도 결정)
4. _호환 매트릭스_ — 클라이언트·서버 버전 조합별 동작 여부

**작성 예 — 클라이언트·서버 버전 호환 매트릭스 (책 §2.8 예시):**

```markdown
| 클라이언트 \ 서버 | v1.0               | v1.1                     | v2.0                 |
| ----------------- | ------------------ | ------------------------ | -------------------- |
| v1.0              | OK                 | OK                       | 일부 기능 동작 안 함 |
| v1.1              | OK (자동 fallback) | OK                       | OK                   |
| v2.0              | ❌ 비정상 동작     | OK (서버 v1.1 호환 모드) | OK                   |
```

이 표는 _고객 지원 부서가 어떤 버전 조합에서 문제가 발생할 수 있는지_ 즉시 확인할 수 있게 합니다.

**최초 개발 시:**

```markdown
## 2.8 Backward Compatibility

본 시스템은 첫 번째 버전이므로 하위 호환성 고려 대상이 없다. `N/A`
```

**3장 Environment 작성 가이드 (공통 도입 — 책 §3장 핵심):**

§3장의 진짜 목적은 _마케팅·QA·CS·고객의 동상이몽 방지_ 입니다. 환경 정의가 부정확하면 다음 일이 _실제로_ 일어납니다 (책 §3장 명시):

- 운영: "GPU 가 필수인 줄 몰라 장비 미구매" → 출시 후 부랴부랴 대응

**3.1 Operating Environment 작성 가이드:**

**작성 모드 선택 (책 §3.1 명시):**

| SW 유형                                   | 작성 방식                                                                           |
| ----------------------------------------- | ----------------------------------------------------------------------------------- |
| _설치형_ (모바일 앱·데스크톱 SW·임베디드) | **최소 사양 / 권장 사양 2 종** 명시 + _각각의 의미_ 정의 (어디까지 동작 보장하는지) |
| _서비스_ (백엔드 API · SaaS)              | **정확한 사양 1 종** (운영팀이 인프라 발주 근거)                                    |

**최소·권장 사양의 _의미_ 를 SRS 에 함께 적기:**

- _"최소 사양"_ — SW 가 _기동되고 기본 기능만_ 동작. 일부 기능은 동작하지 않을 수 있고, 큰 데이터는 처리 못 함.
- _"권장 사양"_ — _모든 기능이 원활히 동작_. 다만 *너무 큰 데이터를 다루는 경우*는 정상 동작 보장 X.
- 의미를 적지 않으면 _"최소 사양에서 모든 기능이 동작 안 한다"_ 항의가 출시 후 매뉴얼·CS 로 폭주합니다.

**OS·런타임 사양 — _구체적이되 광범위 금지_:**

- ❌ "안드로이드 지원" — 너무 막연 (테스트 비용 폭증, 동작 보장 모호)
- ✅ "Android 10 (API 29) 이상, ARMv8 64-bit. _Android 9 이하는 미지원_" — 구체 + 미지원도 명시
- iOS / Web 도 동일: "iOS 15.0 이상", "Chrome 110+ / Safari 16+" 식

**3.5 Test Environment 작성 가이드:**

**책 §3.5 핵심 — 운영 환경 ≠ 테스트 환경:**

**특수 장비·서비스 사전 준비 룰 (책 §3.5 명시):**

| **4.4 Software Interface** | 연동하는 외부 소프트웨어(SDK, DB, 라이브러리) 명시. **DB 테이블/API를 다루는 경우 아래 "DB 스키마 완전성 규칙"과 "API 스키마 완전성 규칙"을 반드시 준수** |
| **4.5 Communication Interface** | 통신 프로토콜(REST, Socket.IO 등), 보안(TLS), 데이터 형식 |
| **4.6 Other Interface** | 위 항목에 해당하지 않는 기타 인터페이스 |

**4.1 작성 필수 포함 항목:**

- **ERD (PostgreSQL 형식)**: 테이블 정의는 PostgreSQL DDL 또는 Mermaid `erDiagram` 형식으로 작성합니다. `*-docs` 레포지토리의 `database/` 디렉터리에 관리합니다

**4장 External Interface Requirements 작성 가이드 (공통 도입 — 책 §4장 핵심):**

§4장의 진짜 목적은 *시스템을 적절히 컴포넌트로 나누고 그 경계를 정의*하는 것입니다. 책의 비유: _항공모함 엔진 — 주위 부품과 어떻게 끼워 맞을지 미리 정한다_.

**내부 vs 외부 인터페이스의 _관점_ (책 §4장 명시):**

- 최상위 SRS 의 _외부 인터페이스_ = 우리 시스템 ↔ 외부 시스템
- 하위 SRS 의 _외부 인터페이스_ = 우리 컴포넌트 ↔ 같은 시스템 내 다른 컴포넌트
- → **같은 인터페이스라도 SRS 의 _레벨_ 에 따라 내부 ↔ 외부 분류가 달라집니다.** 본 SRS 가 어느 레벨인지 (전체 시스템 / 단일 컴포넌트) §1.2 와 §2.1 에서 먼저 명확히.

**컴포넌트 나누기 균형:**

너무 _크면_ 복잡도 흡수 어렵고, 너무 _작으면_ 이해 어려움. 컴포넌트 간 _독립성이 높을수록_ 좋음 (변경 영향 최소화).

**4.1 System Interface 작성 가이드 (책 §4.1 보강):**

위 _4.1 작성 필수 포함 항목_ 절은 Swagger + ERD 링크 룰. 추가로 책 §4.1 의 **인터페이스 정의 시 고려할 13 종 속성**을 Swagger 어디에 매핑할지 정리:

| #   | 속성 (책 §4.1)    | Swagger/OpenAPI 대응 위치                     |
| --- | ----------------- | --------------------------------------------- |
| 1   | 우선순위          | description 에 `(P1)` 표기                    |
| 2   | 데이터 타입       | `type` (string/integer/boolean/object/array)  |
| 3   | 데이터 크기·형식  | `maxLength` / `format` (email/date-time/UUID) |
| 4   | 데이터 단위       | description 에 명시 ("milliseconds", "KRW")   |
| 5   | 데이터 범위       | `minimum` / `maximum` / `enum`                |
| 6   | 발생 시간         | description ("매월 1 일 00:00 KST")           |
| 7   | 발생 빈도         | description + rate limit (HTTP 429)           |
| 8   | 발생 순서         | sequence diagram 또는 description             |
| 9   | 데이터 보안       | `security` (Bearer JWT / API Key)             |
| 10  | 인터페이스 조합   | tag 또는 별도 endpoint 그룹                   |
| 11  | 커뮤니케이션 방법 | HTTP method (REST) / Socket event             |
| 12  | 프로토콜          | servers 의 URL 스킴 (https / wss)             |
| 13  | 에러 처리 방법    | `responses` 4xx/5xx + 커스텀 에러 코드        |

**모든 인터페이스에 13 종 다 적기는 시간 낭비 (책 §4.1 명시):**

- _상황에 맞게_ 필요한 속성만 선별. 단 _우선순위(1)·발생 빈도(7)·에러 처리(13)_ 는 자주 누락되니 의식적으로 점검.
- 외부 표준 문서가 있으면 _링크 참조_ 로 충분 (예: Apple GameKit 가이드 URL).

**내부 인터페이스 분리 옵션 — IRS (Interface Requirements Specification):**

인터페이스가 너무 많으면 SRS 본문이 비대해집니다. _별도 IRS 문서_ 로 분리하고 SRS §4.1 에는 _링크만_ 두는 것도 OK (책 §4.1 명시).

**4.5 Communication Interface 작성 가이드:**

§4.5 는 *통신 프로토콜·보안·데이터 형식*을 정의합니다.

**작성 항목 (책 §4.5 명시):**

1. _사용 표준 프로토콜_ — HTTP/HTTPS / WebSocket / Socket.IO / gRPC / SMTP / FTP / MQTT 등
2. _보안·암호화_ — TLS 버전 (1.2+), 인증서 발급 방식, mTLS 여부
3. _데이터 전송 속도_ — 대역폭 가정, 압축 사용 여부 (gzip / brotli)
4. _동기화 메커니즘_ — Idempotency Key / Sequence Number / Heartbeat / 재시도 정책
5. _방화벽·포트_ — 노출 포트 (예: 443, 80, 8443) + _왜 그 포트인지_

**사용자 정의 프로토콜 사용 시 (책 §4.5 명시):**

- 표준 프로토콜이 아닌 *자체 정의 프로토콜*이 있으면 _별도 문서로 명세 필수_ (메시지 포맷·해석 규칙·에러 처리)
- 표준 프로토콜 사용 시 _RFC 번호 / W3C 사양 / 공식 문서 링크_ 참조로 충분
- _§4.1 과 통합 작성도 OK_ — 통신 인터페이스가 §4.1 System Interface 와 *경계가 모호*하면 §4.1 에 포함. _어디에 적었는지만_ 일관되면 됨.

**작성 예:**

```markdown
## 4.5 Communication Interface

- 프로토콜: HTTPS (TLS 1.3 — Let's Encrypt 인증서, 90 일 자동 갱신)
- 실시간: Socket.IO over WSS (TLS 1.3), Redis Adapter 로 다중 서버 동기화
- 보안: Bearer JWT (Access 1 h + Refresh 14 d)
- 동기화: 메시지 송신에 Idempotency Key 필수 (UUID v4), 3 회 재시도 후 dead letter queue
- 방화벽: 443 (HTTPS/WSS) — 그 외 모두 차단
- 외부 메일 발송: AWS SES (TLS), 발신자 도메인 SPF + DKIM 등록
```

#### 5장 Performance Requirements

전체 기능에 공통 적용되는 성능 요구사항만 기술합니다. 특정 기능의 성능은 7장에 기술합니다.

모든 수치는 구체적으로 측정 가능하게 작성합니다.

- ❌ `사용자가 기다리지 않아야 한다`
- ✅ `95%의 요청이 300ms 이내에 처리되어야 한다`

**5장 작성 가이드 (공통 도입 — 책 §5장 핵심):**

- 성능은 _운영 환경에 따라 달라짐_ → 모든 §5.x 수치는 _§3.1 Operating Environment 의 표준 사양 기준_ 임을 본 절 도입부에 명시
- 적는 *기준*은 프로젝트마다 다름: _시장 요구 수준_ / _목표_ / _검증된 성능_ 중 어느 것인지 표기 (예: "(시장 요구)", "(목표 — 베타 시점 검증)")
- _대안 검토 박스 의무 대상_: 응답 시간·동시 세션·가용성 등의 _수치 결정_ (§"공통 작성 규칙 → 핵심 결정 대안 검토 박스" 참조)

---

**5.1 Throughput 작성 가이드:**

초당 작업 처리 속도. *단위가 도메인마다 다르므로 단위를 명시*해야 합니다.

| 도메인            | 단위 예시                                |
| ----------------- | ---------------------------------------- |
| API 서버          | RPS (Request/sec), TPS (Transaction/sec) |
| DB                | records/sec (INSERT/UPDATE/DELETE)       |
| 데이터 파이프라인 | MB/sec, events/sec                       |
| 미디어/스트리밍   | frames/sec, MB/sec                       |
| 메일·푸시         | mail/sec, push/min                       |

**작성 예 (책 §5.1):**

- `초당 3,000 건의 거래를 처리할 수 있어야 한다 (TPS — 5.1 항목 1)`
- `300 대 카메라에서 전송되는 데이터를 실시간 처리 — 최대 1.2 Gbps`

---

**5.2 Concurrent Session 작성 가이드:**

동시에 연결·처리할 수 있는 세션 수. *세션의 정의*가 도메인마다 다르므로 _세션의 의미를 함께 정의_:

- API 서버: _동시 접속 사용자_ (HTTP keep-alive 기준) / _동시 활성 JWT_ (마지막 1 분 안 요청)
- WebSocket: _동시 연결된 소켓_
- 비디오: _동시 채널 / 동시 시청자_
- 게임 서버: _동시 접속 게임 세션_

**작성 예:**

- `한 게임 서버 인스턴스에 5,000 명이 동시 접속 가능`
- `WSS 채팅 서버는 인스턴스당 10,000 동시 소켓을 유지하며, 인스턴스 N 대 수평 확장`

---

**5.3 Response Time 작성 가이드:**

**책 §5.3 핵심:**

- _모든_ 기능에 대응시간을 적는 것은 아니다 — _민감한 기능만_
- 게임·증권 같이 *실시간성이 핵심인 시스템*은 매우 정밀하게 정의
- _요구되는 대응시간에 따라 아키텍처가 완전히 바뀜_ → 스펙에서 잘 정의해야 함
- _프로토타입으로 미리 검증_ 가능 (책 §5.3 명시)

**분위수 (Percentile) 표기 권장:**

평균(mean)은 *이상치(outlier)*에 가려져 사용자 체감을 못 표현합니다. _p50 / p95 / p99_ 분위수로 표기:

- ❌ `평균 응답 시간 300 ms` — 1% 사용자는 5 초를 기다려도 평균은 OK 로 보일 수 있음
- ✅ `p95 응답 시간 300 ms, p99 응답 시간 1 s` — 사용자 95%·99% 가 체감하는 한계 정의

**작성 예:**

- `사용자가 거래 버튼을 누르면 *p95 0.3 s, p99 1 s* 안에 거래 처리`
- `게임서버는 사용자 커맨드를 *p99 0.05 s* 안에 처리`
- `트랜잭션의 *95% 가 1 초 안에* 처리되어야 한다`

---

**5.4 Performance Dependency 작성 가이드:**

**책 §5.4 핵심 — 트레이드오프 명시:**

§5.1~§5.3 의 수치는 _서로 트레이드오프_ 관계인 경우가 많습니다. 그것을 _명시적으로 적지 않으면_ 운영팀이 _전부 동시 만족 가능한 줄_ 오해합니다.

**자주 발생하는 트레이드오프:**

| 트레이드오프                | 의미                                            |
| --------------------------- | ----------------------------------------------- |
| 동시 세션 ↑ → 응답 시간 ↓   | 동시 접속이 5,000 초과 시 응답 시간 급격히 저하 |
| 동시 세션 ↑ → 최대 처리량 ↓ | 자원 경합                                       |
| 정확도 ↑ → 응답 시간 ↓      | ML 모델 정확도와 추론 속도                      |
| 가용성 ↑ → 비용 ↑           | 99.9% 가용성 = 월 비용 N 배                     |

**작성 예:**

- `동시 세션이 5,000 개를 넘어가면 *p95 응답 시간이 300 ms → 1.5 s 로 저하* 됨. 5,000 초과 운영 시 인스턴스 수평 확장 필요`
- `5.1 의 TPS 와 5.2 의 동시 세션은 *반비례* — 동시 세션 N 배 증가 시 TPS 1/N 배`

---

**5.5 Other Performance Requirements 작성 가이드:**

도메인 특화 성능 요구사항을 적는 곳. *§5.1~§5.4 로 표현 못 하는 것*을 여기로.

**도메인별 작성 예:**

| 도메인          | 추가 성능 요구사항                                                                     |
| --------------- | -------------------------------------------------------------------------------------- |
| AI/ML           | Precision / Recall / F1 / Accuracy / Top-K Hit Rate + _성능 측정용 별도 테스트셋 정의_ |
| 모바일 앱       | 콜드 스타트 시간(p95 ≤ 2 s) / 메모리(평균 ≤ 150 MB) / 배터리(1 시간 사용 시 ≤ 5%)      |
| 프론트엔드      | Web Vitals — LCP ≤ 2.5 s, FID ≤ 100 ms, CLS ≤ 0.1                                      |
| 결제            | 결제 콜백 처리 _Idempotency 보장 100%_ (중복 처리 0 건)                                |
| 미디어 스트리밍 | 버퍼링 발생률 ≤ 1%, 화질 변경 횟수 ≤ 2 회/분                                           |

**책 §5.5 명시 — 성능 측정 방법도 함께:**

- 측정 방법 / 테스트 데이터셋 / 평가 시점 / 평가자 모두 정의해야 _반복 검증 가능_
- 예: "피부암 진단 엔진 — 개발팀에 _공개되지 않는_ 피부암·정상 이미지 1,000 장을 별도 보관해 _베타 / GA 시점에 측정_"

**6장 작성 가이드 (공통 도입 — 책 §6장 핵심):**

**책 §6장 경고 — 비기능을 누락하면 _시스템 전반에 영향, 상당 부분 재개발_:**

- _흔한 실수_: SRS 에 기능 요구사항만 잔뜩 적고 비기능을 빠뜨림
- _비기능은 외부에서 안 들어옴_ — 경험 많은 개발자가 _분석으로 찾아내야_ 함
- _아키텍처에 큰 영향_ → 나중에 _함부로 바꾸기 어려움_ (기능 추가보다 비기능 변경이 _훨씬 어려움_)

**AI 의 행동 룰:**

- AI 가 SRS 초안을 작성할 때 §6.x 의 각 항목을 _기계적으로 점검_ (없으면 `None`/`N/A` 표기 — _항목 자체를 삭제 금지_)
- *발견한 비기능 후보*는 §"공통 작성 규칙 → 핵심 결정 대안 검토 박스" 의무 대상에 포함될 수 있는지 함께 검토

**6.1 Safety 작성 가이드:**

**§6.1 의 의미 (책 §6.1):**

시스템 사용으로 인한 _손실·손해·상해_. 사람 _부상·생명·재산_ 보호. 사무용 SW 는 적지만 _원자력·비행기·CT·은행·자동차·결제·헬스케어_ 등은 필수.

**도출용 5 가지 질문 (책 §6.1 명시 — 자체 검증 체크리스트):**

1. 정상 또는 비정상 동작이 사용자에게 *건강·생명·재산상 피해*를 줄 수 있는가?
2. 그 피해 확률이 얼마나 되는가? (낮으면 §2.6 Assumptions 에 명시)
3. 어떤 *비정상 종료·장애*가 있을 수 있는가?
4. *사용자 실수*가 손실을 일으킬 수 있는가?
5. 비정상 종료 시 *피할 수 없는 피해*는 무엇인가?

**무토 도메인 예시 (책 §6.1 패턴 적용):**

- 결제 — *평소 평균 결제 금액의 100 배 이상 주문*은 추가 확인 1 회 더
- 인증 — 비밀번호 5 회 실패 시 10 분 잠금 (자동 풀림)
- 데이터 — 사용자 계좌·결제 정보는 _지리적으로 분리된 백업_ 실시간 유지
- 채팅 — 외부 로그 서버 연결 끊김 시 _재연결 성공까지 로컬 큐에 저장 후 발송_
- 매칭/추천 — 부적절 콘텐츠 자동 차단 + 사용자에게 신고 절차 고지

**잘못된 예 (책 §6.1 명시):**

- ❌ "본 SW 는 항상 정상 동작해야 한다" — 당연하거나 불가능한 일반론
- ❌ "운석이 떨어져 데이터 손실은 어쩔 수 없다" — _희박한 확률은 굳이 고려할 필요 없음_

**법률·인증 준수 (책 §6.1):**

본 SW 가 따라야 할 _법률·인증 요구사항_ (의료기기 인증·금융감독 규정·개인정보보호법 등) 도 §6.1 에 함께 명시. 그에 따른 _기능 요구사항_ 은 §7 에 추가.

---

**6.2 Security 작성 가이드:**

**§6.2 의 의미 (책 §6.2):**

데이터·개인정보 보호. 악의적 접속·사용·수정·공격으로부터 SW 보호. _"보험과 비슷 — 문제 생기기 전까지 필요성 못 실감"_ (책 §6.2 명시).

**보안 분석 8 항목 (책 §6.2 명시 — 모든 SRS 가 8 종을 한 번씩 점검):**

| #   | 항목                       | 무토 적용 예                                         |
| --- | -------------------------- | ---------------------------------------------------- |
| 1   | Authentication (인증)      | JWT (Access 1 h + Refresh 14 d), OAuth (Kakao/Naver) |
| 2   | Authorization (권한 부여)  | RBAC — Admin/Operator/User 분리                      |
| 3   | Access control (접근제어)  | IP 화이트리스트(어드민), Tenant 격리                 |
| 4   | Non-repudiation (부인방지) | 결제·인증 동의 로그 영구 보존                        |
| 5   | Confidentiality (기밀유지) | DB at-rest 암호화, TLS 1.3 in-transit                |
| 6   | Integrity (무결성)         | DB transaction, payment idempotency key              |
| 7   | Secure coding              | OWASP Top 10 점검, Snyk/Dependabot                   |
| 8   | Web vulnerabilities        | CSRF/XSS/SQLi 방어, CSP 헤더                         |

**보안과 편리의 트레이드오프 (책 §6.2 명시):**

- 보안을 _너무 과도하게_ 요구하면 *편리성·사용성*이 줄어듦
- *꼭 필요한 보안 요구사항*만 적고 _적절한 수준_ 유지
- 예: 일반 사용자 로그인은 비밀번호로 충분, 결제 시점에만 2FA — _행위별 보안 강도 분리_

**잘못된 예 (책 §6.2 명시):**

- ❌ "사용자 암호는 암호화해서 보관해야 한다" — _암호화 방식이 무엇인지_ (bcrypt cost 12 / Argon2id 등) 모호

**작성 예:**

- 사용자가 10 분 이내 연속 5 회 로그인 실패 시 10 분간 추가 로그인 차단
- 모든 어드민 세팅 변경에 대해 _operator id · timestamp · before/after value · IP_ 로그 영구 보존
- 비밀번호는 _Argon2id (memory=64 MB, iterations=3)_ 로 해싱 후 저장

**6.3 System Attributes 작성 가이드 (6.3.1~6.3.5 통합):**

**6.3.1 Availability — 책 §6.3.1:**

- 가용성 _수치 표기_: 24/7 / 99.9% / 99.999% / _사용 시만 가동_ 등 명시
- _가용성 요구사항은 설계에 큰 영향_ — 99.99% 이상은 _별도 standby 시스템_·_즉시 복구_ 등 필요
- _유지보수 시간_ 함께 명시: "0~0:30 패치, 0:30~24:00 가동 → 23.5 h/일 = 약 97.9%"
- 무토 표준 예: API 99.9% (월 다운타임 ≤ 43 분), 채팅 99.5% (월 ≤ 3.6 h)

**6.3.2 Maintainability — 책 §6.3.2:**

- 단순 _"유지보수 쉬워야 한다"_ X — _기술·요구사항으로_ 기술
- 모듈화 수준 / 인터페이스 정의 / 복잡도 한계 / 자동 문서화 / _원격 지원 기능_ / _Crash report 자동 수집(Firebase Crashlytics 등)_
- 무토 표준: NestJS 모듈 단위 / Flutter feature 단위 분리, Crashlytics + Sentry 자동 수집

**6.3.3 Portability — 책 §6.3.3:**

- _호스트 종속 컴포넌트·코드 비율_, 이식 용이한 언어/컴파일러 사용 여부
- 이식 *불필요*도 결정: _"현재 안드로이드만, iOS 지원 계획 없음 → 이식성 미고려"_ 명시
- 무토 모바일: Flutter 단일 코드베이스 — _플랫폼 종속 코드 5% 미만 유지_

**6.3.4 Reliability — 책 §6.3.4:**

- _MTBF (Mean Time Between Failures)_ — 시스템이 _정상 작동되는 평균 시간_
- _신뢰성 ≠ 크래시 0_ — MTBF 가 높다는 것은 *심각한 문제 가능성이 낮다*는 의미
- *시스템 전체 신뢰성 ≤ 컴포넌트 *최소값\*\*\* (체인의 가장 약한 고리 룰)
- _높은 신뢰성은 공짜 X_ — 비용 발생, 아키텍처 반영 필수
- 작성 예: "결제 모듈은 _runtime 동안 crash 발생 시 자동 재시작 + 진행 중 트랜잭션 보존_", "본 시스템은 30 일 정상 동작 보장 + 매월 정기 재기동"

**6.3.5 Remaining Attributes — 책 §6.3.5 7 종 후보:**

이미 위 6장 표에 명시. _해당 프로젝트와 관련 있는 것만 선택_. 관련 없으면 `None`.

**6.4 Logical Database Requirements 작성 가이드:**

**§6.4 의 의미 (책 §6.4):**

DB 의 _논리적 요구사항_. 다음 6 항목 포함:

1. 다양한 기능에서 사용되는 _정보 유형_
2. 정보의 _사용 빈도_
3. _접근 용량_ (예: 일 100만 read / 1만 write)
4. 데이터 _엔티티와 그 관계_ (ERD)
5. _무결성 요구사항_ (FK·CHECK·UNIQUE)
6. _데이터 보존 기간_ (예: 결제 로그 5 년, 채팅 메시지 1 년)

**무토 표준 — §"공통 작성 규칙 → DB 스키마 완전성 규칙" 적용:**

§6.4 에 신규 테이블·기존 테이블 변경을 언급할 때 _컬럼명·타입·기본값·nullable·인덱스·relation 7 종_ 모두 명시 (Prisma 모델 수준). DDL 만 적고 끝나면 마이그레이션 작성자가 추측해야 함.

**작성 양식:**

```markdown
## 6.4 Logical Database Requirements

- ERD: [erd.md (main branch commit `abc1234`)](https://github.com/.../database/erd.md)
- 접근 용량 (v1.0 기준): 일 100 만 read / 1 만 write
- 무결성: 모든 FK 는 `ON DELETE RESTRICT`, 결제 로그는 `IMMUTABLE` (UPDATE/DELETE 금지)
- 보존 기간: 결제 로그 5 년 / 채팅 메시지 1 년 / 분석 이벤트 90 일 후 cold storage 이전
```

**6.5 Business Rules 작성 가이드:**

**§6.5 의 의미 (책 §6.5):**

제품이 따라야 할 _원칙·규칙_ — 법규·산업 표준·회사 내부 규칙·도메인 알고리즘. _기능 요구사항은 아니지만_, 이 규칙을 지키기 위해 특정 기능이 추가됨.

**비즈니스 _규칙_ vs 비즈니스 _요구사항_ (혼동 주의):**

- _규칙_: "주문 금액 $3,000 이상이면 자동 5% 할인 적용" — _항상 지켜야 할 정책_
- _요구사항_: "VIP 사용자에게 추가 할인 쿠폰을 발송한다" — _기능 구현 목표_

**작성 영역 (도메인별):**

| 카테고리        | 무토 예시                                                                  |
| --------------- | -------------------------------------------------------------------------- |
| 법규·인증       | 전자상거래법, 정보통신망법, 청소년 보호법, GDPR (글로벌 진출 시)           |
| 산업 표준       | PCI-DSS (결제), KISA (개인정보), PG 정산 규칙                              |
| 회사 내부 규칙  | 환불 정책 (구매 7 일 내), 무료 체험 1 회 한정, 리뷰 작성 후 24 h 수정 가능 |
| 도메인 알고리즘 | 매칭 점수 공식, 추천 가중치, 큐레이션 우선순위                             |
| 권한·역할       | Admin level 이상은 모든 트랜잭션 로그 조회 가능                            |

**작성 예 (책 §6.5):**

- 고객이 결제에 사용한 신용카드가 없어도 _환불 가능_ (법규 — 전자상거래법)
- 유효기간 3 개월 미만 여권은 _국제선 탑승 불가_ (외부 표준)
- 주문 금액 ≥ $3,000 → _자동 5% 할인_ (회사 정책)

**6.8 Operations 작성 가이드:**

**§6.8 의 의미 (책 §6.8):**

시스템 운영을 위해 _사용자·운영자가 해야 할 작업_. 다음 4 카테고리:

1. _대화형_ 운영 작업 — 운영자가 어드민에서 수동 실행
2. _무인_ 작업 — 스케줄러·크론·Lambda 자동 실행
3. _데이터 처리_ — 배치 집계, ETL, 리포트 생성
4. _백업 및 복구_ — 주기·보존·복구 RTO/RPO

**무토 도메인 예 (책 §6.8 패턴):**

- (대화형) 운영자가 어드민에서 _부적절 콘텐츠 신고 처리_ 일 평균 N 건
- (무인) 매일 04:00 KST _DB 백업 → S3 cold storage_, 30 일 보존
- (무인) 매월 1 일 00:00 KST _전월 결제 정산 리포트 자동 생성·이메일 발송_
- (사용자) 운영자는 _주 1 회 감사 로그 점검_, _월 1 회 알람 규칙 리뷰_

**§4.2 User Interface 와의 관계 (책 §6.8 명시):**

§6.8 의 _대화형 운영 작업_ 은 어드민 UI 의 일부 → §4.2 와 일부 중복될 수 있음. _어디에 자세히 적었는지만_ 일관되면 OK.

**6.10 Internationalization 작성 가이드:**

국제화는 **소스코드·아키텍처 전반에 영향**을 주며, 초기 미반영 시 나중에 추가하려면 *수십·수백 배 비용*이 듭니다. 사업 초기에 *언어 1 종만 지원*하더라도, _향후 확장 계획이 있다면_ 6.10 에서 _지금_ 결정해야 합니다.

**3 가지 핵심 용어 (혼동 주의)**

| 용어       | 발음            | 정의                                                                             |
| ---------- | --------------- | -------------------------------------------------------------------------------- |
| **Locale** | 로케일 (번역 X) | 언어 + 지역의 조합. 예: 미국 영어 `en_US`, 호주 영어 `en_AU`, 한국어 `ko_KR`     |
| **i18n**   | 아이에잇틴엔    | Internationalization. SW 를 *여러 Locale 을 지원할 수 있는 아키텍처*로 만드는 것 |
| **L10n**   | 엘텐엔          | Localization. SW 를 _특정 Locale 에 맞게_ 번역·적응시키는 것                     |

**국제화 표준 6 카테고리 (각각 본 SW 가 지원할지 결정해 명시)**

| 카테고리      | 의미             | 예시                                                  |
| ------------- | ---------------- | ----------------------------------------------------- |
| `LC_MESSAGES` | UI 메시지·문자열 | "결제 완료" → "Payment completed"                     |
| `LC_CTYPE`    | 대소문자 구분    | 언어별 대소문자 규칙                                  |
| `LC_NUMERIC`  | 숫자 표기        | `1,234.56` (US) vs `1.234,56` (DE) vs `1 234,56` (FR) |
| `LC_TIME`     | 날짜·시간 표기   | `01/02/2026` 의 의미가 나라마다 다름                  |
| `LC_COLLATE`  | 문자열 정렬      | 언어별 정렬 순서                                      |
| `LC_MONETARY` | 화폐 표기        | `$1,234.56`, `1.234,56 €`, `￥1,234`                  |

**비표준이지만 자주 누락되는 항목 (해당 시 명시)**

- 이름/주소/전화번호 형식
- 색상·아이콘·이모티콘의 문화적 의미
- 단수·복수 표현 (폴란드어 등은 영어보다 훨씬 복잡)
- 쓰기 방향(아랍어는 RTL), 키보드 배열·IME
- 타임존 / 썸머타임
- 종이 크기·온도·길이·무게·부피 단위

**작성 예**

```markdown
## 6.10 Internationalization

본 시스템은 v1.0 에서 한국어(`ko_KR`)만 지원하나, v2.0(국제 출시) 을 대비해 아래 i18n 카테고리는 _초기부터_ 지원하도록 설계한다.

- ✅ `LC_MESSAGES` — 모든 UI 문자열을 `i18n/{locale}.json` 외부 파일로 분리, 코드에 하드코딩 금지
- ✅ `LC_TIME` — 모든 시간은 Unix Timestamp(ms) UTC 로 저장, UI 출력 시 사용자 타임존으로 변환
- ✅ `LC_MONETARY` — 통화 단위는 ISO 4217 코드(`KRW`, `USD`)로 저장, 표시는 사용자 Locale 기준
- ⚪ `LC_NUMERIC` — v2.0 에서 결정 (TBD: 결정 책임자 @홍길동, 결정 마감 v2.0 킥오프, 영향 §7.x 통계 화면)
- ❌ `LC_CTYPE`, `LC_COLLATE` — 본 SW 는 사용자 입력 정렬·대소문자 비교 기능이 없어 적용 대상 아님 (`N/A`)
- ⚠️ 쓰기 방향(RTL) — 아랍 시장 진출 시 추가 검토. v1.0 범위 외 (`Will Not Do`)
```

#### 7장 Functional Requirements

2장에서 설명한 주요 기능을 **상세하게** 분류합니다.

- 7장의 대분류 제목은 **2.4 Product Functions의 항목과 1:1 매핑**되어야 합니다
- 각 기능 항목은 작업량이 **1~2일** 정도로 산정 가능한 수준으로 세분화합니다
- 기능 번호 체계: `7.1` → `7.1.1` → `7.1.1.1` → `7.1.1.1.1` → `7.1.1.1.1.1`
- 각 기능에 우선순위 `(P1)`, `(P2)`, `(P3)` 표시

**7장 작성 가이드 (책 §7 — 8 가지 기능 분류 방법):**

책 §7 은 기능을 적는 방법 8 가지를 소개합니다. *프로젝트 성격에 맞는 방법*을 선택하되, _회사 통일된 방법_ 을 유지하는 것이 좋습니다.

| #   | 분류 방법                            | 적합한 SW 유형                                         | 예시                                           |
| --- | ------------------------------------ | ------------------------------------------------------ | ---------------------------------------------- |
| 1   | System mode                          | _모드별 동작이 크게 다른_ 제어 시스템                  | 제어 시스템 — 학습/정상/비상 모드              |
| 2   | User class                           | 사용자 클래스별 _기능 세트가 완전히 다른_ SW           | 엘리베이터 — 승객/유지보수/소방관              |
| 3   | Objects                              | _객체 중심 도메인_ (의료·CRM·IoT)                      | 환자 모니터링 — 환자/센서/간호사/의약품 객체별 |
| 4   | Feature                              | _입력→결과 쌍이 명확한_ 기능                           | 전화 — 지역통화/착신전환/컨퍼런스 콜           |
| 5   | Stimulus                             | _외부 자극에 반응하는_ 시스템                          | 항공기 자동 착륙 — 전력 손실/바람/롤 변화      |
| 6   | Response                             | _응답 생성이 핵심인_ 시스템                            | 인사 시스템 — 월급 생성/직원 목록 등 응답 단위 |
| 7   | **Functional hierarchy (계층 구조)** | _일반적인 비즈니스 SW_ — 책 §7 저자가 _가장 자주 사용_ | 일반 앱·웹·백엔드 (기능을 계층적으로 분해)     |
| 8   | **Use case scenario**                | _행위자·이벤트 흐름이 중요한_ 사용자 중심 SW           | 쇼핑몰 — 회원가입/구매/결제/환불 시나리오      |

**무토 기본 패턴 (추천):**

- _기본_: 7번 **Functional hierarchy** (`7.1` → `7.1.1` → `7.1.1.1` 계층)
- _보조_: 8번 **Use case scenario** 를 §7 안 _주요 흐름_ 에 부분 적용 (특히 결제·매칭·신고 같은 _다단계 사용자 시나리오_)
- _모드 또는 사용자 클래스가 정말 다를 때만_ 1번/2번 패턴 도입 (예: 어드민 SRS 와 사용자 앱 SRS 분리)

**작성 시 주의 (책 §7 명시):**

- 각 기능에 _고유 레이블_ 부여 (무토 표준: `7.1.1.1` 체계) — 추적·관리 용이
- _너무 광범위 정의 X_ — 요구사항이 점점 커져 적는 것 자체가 부담이 되는 오류 조심
- _인터페이스(§4)·UI(§4.2) 와 중복 X_ — §7 은 _기능 자체_ 만, 인터페이스 상세는 §4 로

**기능 정의 핵심 — 외부 인터페이스로 정의 가능 (책 §7 4번 Feature):**

각 기능은 _일련의 입력(요청) 및 결과 쌍_ 으로 설명할 수 있어야 합니다. AI 가 §7 의 기능 하나를 작성할 때 _Input · Trigger · Output · Side Effect_ 4 항목을 자체 점검:

```markdown
### 7.2.3 추천 후보 노출 (P1)

- **Input**: 사용자 ID, 현재 위치(lat/lng), 활성 필터 (성별·나이대·관심사)
- **Trigger**: 홈 화면 진입 / _Pull to refresh_ / 하단 스크롤 (페이지네이션)
- **Output**: 추천 후보 카드 목록 (최대 30 명, 점수 내림차순) + 다음 페이지 cursor
- **Side Effect**: 노출 이벤트 기록 (§4.4 — `POST /api/v1/impressions`), 추천 알고리즘 학습 데이터 누적

**비목표 (Will Not Do)** — §"공통 작성 규칙 → 비목표 표기 규약" 참조

- 신규 사용자(가입 7 일 미만)는 본 추천 알고리즘 대상이 아님 — §7.5 콜드 스타트 정책 적용
```

#### 8장 Change Management Process

**§8 의 의미 (책 §8):**

본 SRS 의 *변경 관리 절차*를 정의합니다. 책에서는 _필수 챕터가 아니라_ 변경 관리의 중요성을 강조하기 위해 필요 시 추가하라고 합니다. 무토에서는 **베이스라인 v1.0 이상 SRS 는 권장**합니다.

**작성 항목:**

| 항목                       | 무토 표준                                                                         |
| -------------------------- | --------------------------------------------------------------------------------- |
| 변경 관리 프로세스 링크    | 사내 위키 / Notion 의 _변경 관리 정책 페이지_ URL                                 |
| CCB (Change Control Board) | 본 SRS 변경 시 검토·승인할 사람 명단 (PM + 백엔드/모바일/프론트 리드 + QA 리드)   |
| 변경 분류                  | _Minor_ (오타·문구 정리, CCB 패스 가능) vs _Major_ (요구사항 추가·변경, CCB 필수) |
| 영향 평가 의무             | 변경 시 _영향 받는 §·DBML·Swagger·TCL·일정_ 명시                                  |
| 추적 도구                  | GitHub PR + Jira 이슈 키 연결, _Decision Log_ (Appendix A) 한 줄 추가             |

**작성 예:**

```markdown
## 8 Change Management Process

- 변경 관리 정책: [Munto 변경 관리 정책 (Notion)](https://www.notion.so/.../change-mgmt-policy)
- CCB: @PM*홍길동, @BE*김민지, @MOB*박철수, @WEB*이수진, @QA\_정수민
- 분류: Minor (문구·타이포) / Major (요구사항·아키텍처·NFR 수치 변경)
- 변경 절차:
  1. GitHub PR 생성 + 영향 평가 기재 (변경 분류 / 영향 받는 §·DBML·Swagger·TCL·일정)
  2. Major 의 경우 CCB 전원 review·approve 필수
  3. Merge 시 변경 이력 표 갱신 + Appendix A Decision Log 1 줄 추가
  4. 베이스라인 변경이면 GitHub release tag (`v1.x`) 갱신
```

| 안정적으로 운영한다           | 가용성 99.9% (월 다운타임 43분 이하)를 목표로 한다       |

### 기호·수식 선행 정의 원칙

문서에 수학 기호·수식이 등장하는 SRS(통계·확률·랭킹·머신러닝 관련 도메인)는 **독자가 해당 기호를 처음 만나는 시점에서 의미를 알 수 있어야** 합니다.

> 본 절은 수식이 등장하는 도메인에 한정 — 채팅·결제·이메일 등 _수식이 없는 도메인은 N/A_. 다만 본문에 처음 등장하는 비표준 식별자(약어·상태값·외래어 등)의 1.4 등록 판정 원칙은 §"용어·약어(1.4) 작성 원칙" 이 전 도메인 공통입니다.

- 1.3 등 이른 섹션에서 수식 예시로 기호가 등장하면, **해당 위치에서 간략 설명**하거나 `(1.4 참조)` 링크를 명시
- **1.4 용어표에 모든 기호를 정의**: 약어뿐 아니라 수학 변수도 포함
- 7장 등 후반 섹션에서만 상세 정의가 나오는 경우, 1.4 용어표에 간략 정의 + 상세 섹션 참조를 병기

> _아래 예시 도메인: 추천 알고리즘 시스템 (가상)_ — 결제 도메인이라면 `α₀, β₀` 대신 `수수료율 r, 할인율 d` 식으로 치환됩니다.

```markdown
<!-- ❌ 나쁜 예: 1.3에서 기호가 설명 없이 등장 -->

우선순위 점수 = α₀ × k + β₀ × n

<!-- ✅ 좋은 예: 기호 등장 시 즉시 설명 또는 참조 -->

우선순위 점수 = α₀ × k + β₀ × n (각 기호 정의는 1.4 참조)

<!-- ✅ 1.4 용어표 -->

| 기호 | 정의                      | 상세       |
| ---- | ------------------------- | ---------- |
| α₀   | 초기 가중치 (기본값: 0.5) | 7.2.3 참조 |
| β₀   | 보정 계수 (기본값: 0.3)   | 7.2.3 참조 |
| k    | 활성 사용자 수            | 7.2.3 참조 |
| n    | 샘플 크기                 | 7.2.3 참조 |
```

### DB 스키마 완전성 규칙

SRS에서 신규 테이블이나 기존 테이블 변경을 언급하는 경우, **다른 개발자가 바로 마이그레이션을 작성할 수 있는 수준**으로 정의해야 합니다.

**신규 테이블 필수 정의 항목:**

| 항목          | 필수 | 설명                                                            |
| ------------- | ---- | --------------------------------------------------------------- |
| 컬럼명        | ✅   | camelCase (Prisma 컨벤션)                                       |
| 타입          | ✅   | Prisma 타입 (`String`, `Int`, `DateTime`, `Boolean`, `Enum` 등) |
| 기본값        | ✅   | `@default(...)` 또는 "없음" 명시                                |
| nullable 여부 | ✅   | `?` 붙이는지 여부                                               |
| 제약조건      | ✅   | `@unique`, `@id`, `@relation` 등                                |
| 인덱스        | ✅   | `@@index([...])`, `@@unique([...])`                             |
| relation      | ✅   | 다른 모델과의 관계 (`@relation`)                                |

**기존 테이블 변경 시 필수 정의 항목:**

| 항목              | 필수 | 설명                                                |
| ----------------- | ---- | --------------------------------------------------- |
| 추가/변경 컬럼명  | ✅   |                                                     |
| 타입              | ✅   |                                                     |
| 기본값            | ✅   | 기존 데이터 마이그레이션 영향 명시                  |
| nullable 여부     | ✅   |                                                     |
| 마이그레이션 전략 | ✅   | 기존 row 처리 방법 (예: "기존 row는 `null`로 유지") |

> _아래 예시 도메인: 추천 알고리즘 시스템 (가상)_ — 결제 도메인이라면 `PaymentLog`, 채팅이라면 `ChatMessage` 등으로 치환됩니다. 핵심은 *어떤 도메인이든 컬럼·타입·기본값·nullable·인덱스·relation 7 종이 모두 명시*되는 것입니다.

```prisma
// ✅ 좋은 예: 바로 마이그레이션 작성 가능한 수준
model UserAlgorithmScore {
  id        Int      @id @default(autoincrement())
  userId    Int      @unique
  score     Float    @default(0.0)
  updatedAt DateTime @updatedAt
  user      User     @relation(fields: [userId], references: [id])

  @@index([userId])
}

// ❌ 나쁜 예: 표에 컬럼명만 나열
// | 테이블명 | 설명 |
// | UserAlgorithmScore | 사용자 알고리즘 점수 |
```

### API 스키마 완전성 규칙

SRS에서 API(신규 또는 기존)를 언급하는 경우, **구현자와 QA가 바로 작업할 수 있는 수준**으로 정의해야 합니다.

**API 정의 필수 항목:**

| 항목               | 필수 | 설명                                      |
| ------------------ | ---- | ----------------------------------------- |
| HTTP Method + URL  | ✅   | `POST /api/v1/impressions`                |
| Request Body/Query | ✅   | 필드명, 타입, 필수 여부, 제약조건         |
| Response Body      | ✅   | 성공 시 JSON 스키마 (필드명, 타입)        |
| 상태 코드          | ✅   | 200, 201, 400, 401, 403, 404, 409, 500 등 |
| 에러 코드          | ✅   | 커스텀 에러 코드 + 메시지 키              |
| 인증               | ✅   | `Bearer JWT` 필수 여부                    |

> _아래 예시 도메인: 추천 시스템의 노출 기록(impression) API (가상)_ — 결제 도메인이라면 `POST /payments`, 채팅이라면 `POST /messages` 등으로 치환됩니다. 핵심은 *어떤 도메인이든 method+URL / request / response / 상태 코드 / 에러 코드 / 인증 6 종이 모두 명시*되는 것입니다.

```yaml
# ✅ 좋은 예: Swagger/OpenAPI 수준
POST /api/v1/impressions:
  summary: 노출 기록 생성
  security:
    - BearerAuth: []
  requestBody:
    required: true
    content:
      application/json:
        schema:
          type: object
          required: [targetUserId, type]
          properties:
            targetUserId:
              type: integer
            type:
              type: string
              enum: [CARD, PROFILE, RECOMMENDATION]
  responses:
    201:
      description: 생성 성공
      content:
        application/json:
          schema:
            type: object
            properties:
              id: { type: integer }
              createdAt: { type: integer, description: 'Unix timestamp (ms)' }
    400:
      description: 잘못된 요청
      content:
        application/json:
          schema:
            $ref: '#/components/schemas/ErrorResponse'
    401:
      description: 인증 실패

# ❌ 나쁜 예: 표에 URL만 나열
# | API | 설명 | 상태 |
```

| 필수 항목          | 설명                                                                                    |
| ------------------ | --------------------------------------------------------------------------------------- |
| **미결 이유**      | 왜 지금 결정할 수 없는가 (정보 부족 / 외부 의존 / 운영 데이터 필요 등)                  |
| **결정 책임자**    | 최종 결정을 내릴 사람 (`@이름`)                                                         |
| **결정 마감 시점** | 절대 시점("2026-07-01") 또는 상대 시점("런칭 후 4 주")                                  |
| **영향 받는 섹션** | 이 결정이 확정되면 본 문서의 어떤 섹션·하위 산출물(DBML·Swagger·UI)이 갱신되어야 하는가 |

```markdown
<!-- ✅ 허용: 4 항목 전부 명시 -->

- **스팸 방지 정책** (TBD)
  - 미결 이유: 런칭 후 실사용 패턴(IP/디바이스/속도) 데이터가 필요하다
  - 결정 책임자: @홍길동 (백엔드 리드)
  - 결정 마감 시점: 런칭 후 4 주
  - 영향 받는 섹션: 6.2 Security / 7.3 신고·차단 / Swagger `POST /reports`
  - 관련 이슈: DEVT-000

<!-- ❌ 불허용 (이유·책임자·시점·영향 누락) -->

- 스팸 방지 기능이 필요하다고 생각하지만 구체적인 방안이 마련되지 않음

<!-- ❌ 불허용 ("나중에 정함" 류) -->

- 인증 방식: TBD
```

- 베이스라인(v1.0) 통과 시점에 **TBD 항목 목록**을 별도 표로 정리하여 변경 이력에 첨부합니다.

### 해당 없는 항목 처리 — `None` vs `N/A` 구분

항목은 삭제할 수 없습니다. 내용이 없는 경우 아래 표기를 **정확히 구분**해서 사용합니다.

| 표기                         | 의미                                                               | 사용 예                                              |
| ---------------------------- | ------------------------------------------------------------------ | ---------------------------------------------------- |
| `None`                       | 항목은 적용 대상이지만 **이번 프로젝트에서 해당되는 내용이 없다**  | 1.7.3 Patent — 특허 출원 가능 아이디어가 실제로 없음 |
| `N/A`                        | **항목 자체가 본 프로젝트에 적용되지 않는다** (검토 자체가 무의미) | 백엔드 전용 프로젝트의 4.2 User Interface            |
| `N/A(기존과 동일)`           | 기존 문토 환경과 완전히 동일하여 별도 정의가 필요 없음             | 3.1 Operating Environment                            |
| `N/A(기존 배포 환경과 동일)` | 기존 배포 환경과 완전히 동일                                       | 3.3 Distribution Environment                         |

> **`None` 과 `N/A` 의 의미가 다릅니다.** `None` 은 "확인했고 없다", `N/A` 는 "확인 자체가 무관하다" 입니다. 잘못 쓰면 리뷰어가 누락 여부를 판단할 수 없습니다.

> 항목을 **삭제하면** 독자가 "누락된 것인지, 해당 없는 것인지" 판단할 수 없습니다. 삭제 금지.

**상위 항목 축약 규칙**: 하위 항목이 **전부** `None` 또는 `N/A` 계열인 경우, 하위 항목을 개별 기재하지 않고 **상위 항목에 `N/A`를 한 번만 기재**해도 됩니다. 이 경우 하위 항목이 모두 해당 없음으로 간주됩니다.

```markdown
<!-- ✅ 축약 가능: 하위 항목 전부 N/A -->

### 3.1 Operating Environment

N/A

<!-- ❌ 축약 불가: 하위 항목 중 일부만 N/A -->

### 3.1 Operating Environment

#### 3.1.1 Hardware Environment

N/A(기존과 동일)

#### 3.1.2 Software Environment

Node.js 18.15.0, PostgreSQL 15
```

### 용어 표기

- 기술 용어는 **영어 원문** 표기, 최초 등장 시 한글 병기: `Socket.IO(웹소켓)`
- 약어는 최초 등장 시 전체 용어 병기: `MSA(Microservice Architecture)`
- 시간 형식: Unix Timestamp (밀리초)
- 필드명: camelCase

### 비목표(Will Not Do) 표기 규약

**"하지 않는다"는 결정은 "한다"는 결정만큼 중요합니다.** 비목표를 명시하지 않으면 리뷰어·구현자·QA 가 각자 추정해서 일하게 됩니다.

비목표는 **누락 위험이 높은 두 위치**에 명시합니다.

| 위치                                | 무엇을 적는가                                                                                         |
| ----------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **1.2 Product Scope 마지막 1 문단** | 프로젝트 전체 차원의 비목표 (예: "본 프로젝트는 채팅·결제 기능을 다루지 않는다")                      |
| **7.x 각 기능 끝**                  | 해당 기능 차원의 비목표 (예: "이 추천 알고리즘은 신규 사용자(가입 7 일 미만)는 대상으로 하지 않는다") |

```markdown
<!-- ✅ 좋은 예 -->

### 7.2 추천 알고리즘 (P1)

...본문...

**비목표 (Will Not Do):**

- 신규 사용자(가입 7 일 미만)는 본 알고리즘의 대상이 아니다. 별도 콜드 스타트 정책(7.5)을 따른다.
- 비활성 사용자(30 일 이상 미접속)는 후보군에서 제외하며, 본 알고리즘으로 점수를 재계산하지 않는다.

<!-- ❌ 나쁜 예: 비목표 절 자체가 없음 → 구현자가 "신규/비활성 처리"를 임의 결정 -->
```

- 비목표는 **부정형으로 명확히** 작성합니다. "~ 고려하지 않는다", "~ 대상이 아니다", "~ 본 프로젝트 범위 밖이다".
- 단순 누락(쓰지 않음)과 비목표(의도적으로 제외)는 다릅니다. 의도적으로 제외한 것은 반드시 비목표로 명시합니다.

### 논의 기록 (Decision Log) — 핵심 결정의 근거 보존

**"왜 이렇게 결정했는가"가 사라지면, 6 개월 뒤 같은 논의를 반복합니다.** SRS 본문은 결론만 적고, 그 결론에 도달한 과정을 별도 **Decision Log** 에 보존합니다.

**기록 대상**: 다음 중 하나라도 해당하면 Decision Log 에 1 줄을 추가합니다.

1. 1.2 / 2.x / 6.x / 7.x 에서 **여러 안 중 하나를 선택한 결정** (예: "JWT vs 서버 세션 → JWT 채택")
2. 외부 시스템·라이브러리·프로토콜 선택 (예: "WebSocket vs Socket.IO → Socket.IO")
3. 비기능 요구사항의 수치 근거 (예: "응답 시간 300 ms 채택 근거")
4. 비목표(Will Not Do) 로 옮긴 항목 (왜 빼기로 했는가)

**위치**: SRS 본문 끝(7장 뒤)에 부록 `Appendix A. Decision Log` 로 추가합니다.

```markdown
## Appendix A. Decision Log

| 일시 | 결정 사항 | 채택안 | 비교 대안 | 채택 이유 | 결정자 | 관련 섹션 |
| --- | --- | --- | --- | --- | --- | --- |
| 2026-05-12 | 인증 방식 | JWT (Access 1 h + Refresh 14 d) | 서버 세션 / OAuth 위임 | MSA 구조에서 무상태 검증 필요 | @홍길동 | 6.2 |
| 2026-05-13 | 메시지 동기화 | Socket.IO + Redis Adapter | 순수 WebSocket / SSE | 재연결·룸 관리 비용 절감 | @김민지 | 4.5, 7.4 |
| 2026-05-14 | 채팅 기능 비포함 | Will Not Do | — | 본 프로젝트 범위 초과, 별도 chat-backend 진행 | @홍길동 | 1.2 |
```

- 결정이 **번복**되면 새 줄을 추가하고, 이전 결정 줄에는 `(2026-06-01 번복, A.5 참조)` 와 같이 주석을 답니다. 이전 줄을 지우지 않습니다.
- 베이스라인 v1.0 통과 시점의 Decision Log 항목 수를 기록하면, 변경 관리(CCB) 추적이 쉬워집니다.

### 핵심 결정 — 대안 검토 박스 의무화

**AI 가 처음 제안한 안을 그대로 통과시키지 않기 위한 강제 장치입니다.** 다음에 해당하는 "핵심 결정" 에는 본문 안에 **대안 검토 박스**를 반드시 포함합니다.

**대안 검토 박스 의무 대상**:

- 인증·세션·암호화 방식
- 통신 프로토콜 (REST/gRPC/WebSocket/Socket.IO 등)
- 데이터 동기화 전략 (Polling/SSE/Push 등)
- DB 모델링의 분기점 (정규화 vs 비정규화, 단일 테이블 상속 vs 분리)
- 외부 시스템·SDK 선택
- 비기능 요구사항의 수치 결정 (응답 시간·동시 세션·가용성 등)

```markdown
> **🔍 대안 검토 — 인증 방식**
>
> - **채택안**: JWT (Access 1 h + Refresh 14 d)
>   - 장점: MSA 무상태 검증, 모바일/웹 통합 용이
>   - 단점: 토큰 강제 무효화 비용, 로테이션 복잡도
> - **대안 1**: 서버 세션 (Redis 기반)
>   - 장점: 즉시 무효화, 단순한 멘탈 모델
>   - 단점: MSA 전 노드가 동일 Redis 의존 → 단일 장애점
> - **대안 2**: 외부 OAuth 위임 (Auth0)
>   - 장점: 보안 운영 부담 위임
>   - 단점: 월 비용·종속성, 신원 검증 정책 외부화
> - **선정 이유**: 본 프로젝트는 MSA 구조이며 자체 정책(닉네임·연령 검증)이 필요. 운영 비용 최소화 우선.
> - **재검토 조건**: 동시 접속 5 만 초과 시 / 외부 SSO 도입 요구 발생 시
```

- 최소 **대안 2 개**를 비교합니다 ("대안 없음 — 표준이므로" 식의 한 줄 통과 금지).
- 장점·단점·운영 비용·재검토 조건을 모두 포함합니다.
- 박스 내용은 **Appendix A. Decision Log** 의 해당 줄과 1:1 로 연결되어야 합니다.

### AI 작성물 검증 책임 — "통과 = 인수" 원칙

**AI 가 작성한 산출물이라도, 사람이 한 번 통과시키면 그 시점부터 책임은 통과시킨 사람에게 있습니다.**

| 행동                                      | 책임 소재                                |
| ----------------------------------------- | ---------------------------------------- |
| AI 가 SRS·DBML·Swagger·UI·TCL 초안을 생성 | (AI 출력은 가설일 뿐)                    |
| 사람이 PR/리뷰에서 ✅ 또는 LGTM 표기      | **그 사람이 인수자(reviewer-of-record)** |
| 베이스라인 v1.0 설정                      | 인수자 명단·일시가 변경 이력에 기록됨    |

**리뷰어 자기점검 — 통과 직전 5 가지 질문 (모든 사람 리뷰에 적용)**

> 1. 이 문서/산출물 전체를 **처음부터 끝까지** 읽었는가? (스크롤만 내리지 않았는가)
> 2. 본문에 등장하는 비표준 용어 중 **이해관계자 기준 모를 만한 용어**의 뜻을 알고 있는가? 모르면 1.4 와 사내 공유 용어집에서 확인했는가? (모두가 다 아는 용어는 점검 대상에서 제외)
> 3. **Why** 가 명시되어 있는가? (단순히 What 만 적혀 있는 게 아닌가)
> 4. **누락된 케이스가 있는가?** (에러·경계값·동시성·권한·롤백)
> 5. **다른 합리적 대안**이 있는데, 채택안의 우위 근거가 명확한가? 없다면 대안 검토 박스를 요청했는가?

- 5 가지 중 단 1 개라도 "아니오 / 모름" 이 있으면 **통과 보류**합니다.
- 모르는 용어를 발견하면, 통과 전에 작성자에게 **질문하는 것이 의무**입니다. "AI 가 썼으니 맞겠지" 식의 묵시적 통과는 명백한 책임 위반입니다.
- AI 가 1 안만 제시한 경우, "왜 이게 최선인가" 와 "어떤 안과 비교했는가" 를 작성자에게 요청합니다. 답이 없으면 통과 보류합니다.

### 용어·약어(1.4) 작성 원칙

§1.4 는 **본 문서를 읽는 이해관계자가 모를 만한 용어·약어를 설명**하는 곳입니다. _모든_ 용어를 다 적는 곳이 아닙니다. 너무 많이 적으면 핵심이 묻히고, 너무 적게 적으면 독자가 *엉뚱한 결과*로 이해합니다.

**판정 기준 — 1.4 에 등록할지 5 가지 질문**

1. 본 문서의 **이해관계자(개발자·기획자·운영·마케팅·경영진 등)** 중 일부가 모를 가능성이 있는가?
2. 같은 단어가 **다른 의미로 쓰이는 분야**가 있어서 _엉뚱한 해석_ 위험이 있는가? (예: "센터" — 운영센터 vs 동호회 센터)
3. 본 문서 **전반(여러 챕터)**에 등장하는가? (특정 챕터 안에서만 1~2 회 쓰이고 그 챕터를 보는 사람만 알면 충분하면 1.4 등록 불필요 — 챕터 안에서 1 번 설명)
4. 사내·팀 안에서 정의가 갈리는 모호한 용어인가? (예: "활성 사용자" 의 기준)
5. AI 가 작성한 초안에 등장하는데 출처·정의가 불분명한가?

→ 위 5 개 질문 중 **하나라도 YES** 면 1.4 에 등록. 모두 NO 면 등록 불필요.

**1.4 에 적지 않는 것 (적으면 노이즈)**

- 모두가 뻔히 아는 용어: `TCP/IP`, `HTTP`, `JSON`, `UUID`, `URL`, `SQL`
- 도메인 표준 약어 중 본 문서 이해관계자 전원이 아는 것 (예: 백엔드 전용 SRS 의 `JWT`)
- 4 장 시스템 인터페이스 내부에서만 쓰는 알고리즘 전문 용어 (그 챕터를 읽는 개발자만 알면 충분)

**중복 작성 회피 — 공유 용어 사전 활용**

같은 용어가 여러 SRS·DBML·Swagger 에 반복 등장합니다. 매번 1.4 에 새로 적는 대신:

1. **사내 공유 용어집(Notion / 위키)** 을 한 곳에 둡니다 (회사 차원의 단일 진실 공급원).
2. SRS 의 1.4 에는 **해당 용어집 페이지 링크**를 적습니다.
3. *본 문서에 특화된 용어*만 1.4 표에 직접 정의합니다.

```markdown
## 1.4 Terms and Abbreviations

> 일반 사내 용어는 [Munto 용어집](https://www.notion.so/.../glossary) 을 참조한다. 본 절에는 _본 문서에 처음 도입되는 용어_ 와 _본 문서 맥락에서 특별히 정의가 필요한 용어_ 만 정의한다.

| 용어         | 정의                                                                            | 본문 참조 |
| ------------ | ------------------------------------------------------------------------------- | --------- |
| 큐레이션     | 운영자가 후보 사용자 중 일부를 수동 선별해 추천 풀에 올리는 행위                | 7.2.4     |
| SUBMITTED    | 신청서가 사용자에 의해 제출되었으나 운영자 검토 전 상태 (본 워크플로 전용 정의) | 7.3.1     |
| α₀, β₀, k, n | (추천 알고리즘 도메인에 한정) 7.2.3 의 점수 공식에서 사용하는 가중치·변수       | 7.2.3     |
```

**나쁜 예 — 1.4 가 과적되는 패턴**

- 본문 전반에 등장하는 모든 단어를 기계적으로 1.4 에 등록 → 100 개 행으로 비대해져 정작 중요한 용어가 묻힘
- `JWT`, `TCP/IP`, `JSON` 같은 _모두 아는 용어_ 까지 등록 → 노이즈
- 4 장 안에서만 쓰이고 4 장을 보는 개발자만 알면 충분한 용어를 1.4 에 등록 → 마케팅·경영진이 보는 위치에 불필요한 정보

**AI 작성 시 — 자동 등록보다 *판정*이 먼저**

AI 가 본문을 작성하면서 비표준 용어를 _모두 자동_ 1.4 에 등록하는 방식은 1.4 를 과적시킵니다. 대신 AI 는 ① 후보 목록만 뽑고, ② 위 5 가지 질문으로 판정한 결과 등록 대상만 1.4 에 추가하고, ③ 등록 제외 용어는 _제외 사유_ 와 함께 변경 이력에 1 줄 남깁니다 (예: "`JSON`, `JWT` — 이해관계자 전원이 아는 표준 용어로 1.4 등록 제외").

> **참고**: 본문이 3 페이지를 넘는 SRS 인데 1.4 가 비어 있다면 _모든 용어가 다 등록 불필요했는가_ 를 의심해야 합니다. 빈 1.4 자체는 금지가 아니지만, 판정을 누락한 결과로 빈 경우가 훨씬 많습니다. 리뷰어는 _판정 흔적_(예: 변경 이력의 _등록 제외 사유_) 이 있는지 확인합니다.
