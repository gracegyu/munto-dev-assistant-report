## 1. 이 문서가 다루는 것

스펙은 *내용* 만큼 *표현 방식* 이 중요합니다. 같은 사실이라도 어떻게 적느냐에 따라 *오해의 양·검토 시간·재작업 비용* 이 달라집니다. 이 문서는 AI 에이전트가 스펙 *초안 문장을 만들 때* 또는 *사람이 쓴 초안을 다듬을 때* 적용할 *문장 단위 룰* 모음입니다.

**AI 에이전트가 이 문서를 읽어야 하는 시점:**

1. SRS / One Pager 문장을 *최초로 생성* 할 때
2. 사용자가 *"이 문장 좀 더 명확하게"* 라고 요청할 때
3. spec-reviewer 가 *문장 품질을 지적* 할 때
4. 인터페이스(§4 · §7)를 *상세 수준에 맞게 정의* 할지 결정할 때

> spec-standard.md 의 *측정 가능성·모호어 금지·TBD·N/A·Will Not Do* 룰과 **중복되는 부분은 짧게 cross-link** 하고, 여기는 *문장 변환 메커니즘* 위주로 적습니다.

## 2. 가독성을 높이는 10 가지 룰

| # | 룰 | 핵심 |
|---|----|------|
| 1 | **문장을 쉽게 적자** | 누가 읽어도 *한 번에* 이해되는 문장. 어려운 단어 자랑 금지 |
| 2 | **흐름이 자연스럽게** | *핵심 → 부연 설명* 순. 주저리 X, 목록형 ✓ |
| 3 | **다이어그램·표·그림 활용** | 글자만 가득한 페이지 = *읽기 비용 폭증*. 핵심 정보는 표/도식 |
| 4 | **찾기 쉽게** | 모든 SRS 가 *같은 챕터 번호 = 같은 의미* 가 되도록 (spec-standard.md 의 고정 항목 구조 룰과 동일) |
| 5 | **독자 수준에 맞춰** | 1~2 장 = 일반 용어, 4~7 장 = 기술 용어 OK. 한 문서 안에서도 *섹션마다 톤이 달라야 함* |
| 6 | **하향식 (Top-down)** | 1 장 = 가장 추상, 7 장 = 가장 구체. *처음부터 서술식 상세* 금지 |
| 7 | **최대한 적게 적자** | 작성자가 1 분 더 써서 *읽는 사람 30 초씩* 절약 → 누적 효과 큼 |
| 8 | **일관된 포맷 룰** | 글자 크기·강조·표 양식 통일. spec-standard.md 의 *공통 작성 규칙* 따름 |
| 9 | **링크 활용** | 외부 참조 문서는 *복사 X, 링크 ✓*. 단, *링크 깨짐 방지* 책임은 작성자에게 |
| 10 | **지루하지 않게** | 사내 SRS 라면 *적절한 유머 OK*. (예: *"이 기능은 OOO 가 책임진다고 했음"*) — 단, *외부 제출용은 예외* |

**AI 적용 룰:**

- AI 는 *룰 1~3* 을 *항상* 적용해야 합니다.
- *룰 5* (독자 수준)는 *섹션 번호로 자동 판정*:
  - **1~2 장** → 일반 용어, *PM·기획·경영진이 이해 가능* 한 톤
  - **3 장** → 인프라·DevOps 용어 OK
  - **4·6·7 장** → 개발자 용어 OK
  - **5 장** → 측정 가능한 *숫자 위주*, 부드러운 톤
- *룰 10* (유머)는 AI 가 *먼저 넣지 말고* 사용자가 요청 시에만.

## 3. 문장 작성 — Do/Don't 표

### 3.1 기본 문장형

> *"이 소프트웨어는 OOO 를 해야 한다"* 가 스펙 문장의 기본형 (책 §9.3)

| 영어 | 한국어 권장 | 의미 |
|------|------------|------|
| shall / must / have to | **~해야 한다** | 필수 (P1 / Must) |
| should | **~하는 것이 좋다** | 권장 (P2 / Should) |
| may / can | **~할 수 있다** | 선택 (P3 / May) |
| shall not / will not | **~하지 않는다** | 비목표 (Will Not Do) |

요구사항의 중요도는 *동사 강조가 아닌 우선순위 표기* 로 명확화합니다 (spec-standard.md *우선순위 정의*).

### 3.2 능동태 vs 수동태

| ❌ 수동태 | ✅ 능동태 |
|----------|----------|
| 주문 시 주문 내역은 *즉시 기록된다* | 주문 시 *서버는* 즉시 주문 내역을 *DB 에 기록한다* |
| ABC 라는 기능이 *요구되었다* | *김철수 부장이* ABC 라는 기능을 *요구하였다* |

**왜 능동태?** 수동태는 *주어가 누락되기 쉬워* 책임자가 모호해집니다.

### 3.3 간결한 문장 (한 문장 = 한 의미)

| ❌ 장황 | ✅ 분리 |
|--------|--------|
| 사용자가 결제를 시도하면 시스템은 카드 검증 → 한도 확인 → PG 호출 → 응답 수신 → DB 기록 → 푸시 발송 → 영수증 메일을 한 트랜잭션 안에서 처리한다 | 1) 사용자가 결제를 시도하면 시스템은 카드 검증·한도 확인을 수행한다. 2) 두 단계가 모두 성공하면 PG 를 호출한다. 3) PG 응답을 수신하면 결제 결과를 DB 에 기록한다. 4) DB 기록 성공 후 푸시·영수증 메일을 *비동기* 로 발송한다 |

### 3.4 부정문 vs 긍정문

| ❌ 부정문 우선 | ✅ 긍정문 우선 |
|---------------|---------------|
| 10 개 이상은 지원하지 않는다 | 10 개까지 지원한다 |
| 비회원은 결제할 수 없다 | 회원만 결제할 수 있다 |

긍정문으로 표현이 어색하면 부정문 *덧붙이기* 도 가능:  *"10 개까지 지원한다. 11 개 이상의 요청은 거부한다."*

### 3.5 이유 적기 (선택적, 효과 큼)

같은 사실이라도 *이유* 가 있으면 *유지보수·재사용성* 이 폭증합니다.

| 이유 없음 | 이유 있음 |
|-----------|----------|
| 이 소프트웨어는 Java 로 개발한다 | *향후 Linux 포팅을 위해* 이 소프트웨어는 Java 로 개발한다 |
| 결제는 Toss Pay 만 지원한다 | *PG 정산 단순화·수수료 협상력 확보를 위해* 결제는 Toss Pay 만 지원한다 |

**모든 문장에 이유를 적지 않습니다.** *결정이 뒤집힐 가능성이 있는 경우* 에만. 핵심 결정은 spec-standard.md *Decision Log* 룰을 따릅니다.

### 3.6 서술식으로 (명사 나열 금지)

한국어는 명사만 나열해도 *대충 통하지만* 스펙에서는 *오해 폭증* 입니다.

| ❌ 명사 나열 | ✅ 완전한 문장 |
|-------------|---------------|
| 로그 기록 | *서버는* 로그를 *파일로* 기록해야 한다 |
| 사용자 패스워드 암호화 | *사용자 패스워드를* bcrypt 로 *암호화하여 DB 에* 저장한다 |
| 결제 실패 처리 | *결제가 실패하면* 사용자에게 *실패 사유 문구를 표시하고*, 카트 화면으로 돌아간다 |

### 3.7 주어 생략 금지

한국어는 주어 생략이 자연스럽지만 스펙에서는 *치명적* 입니다.

| ❌ 주어 생략 | ✅ 주어 명시 |
|-------------|-------------|
| 로그를 기록하고 확인할 수 있어야 한다 | *서버는* 로그를 기록한다. *관리자는* 어드민 페이지에서 로그를 확인할 수 있다 |
| 결제 후 푸시를 보낸다 | *결제 모듈이 결제 성공 이벤트를 발행* 하면 *Notification 모듈이* 결제자에게 푸시를 보낸다 |

### 3.8 측정 가능 표현 (spec-standard.md *측정 가능성 원칙* 참조)

자세한 룰은 spec-standard.md 에 있으니, 여기서는 *변환 메커니즘* 만:

| 모호어 | 변환 키워드 | 결과 |
|--------|------------|------|
| 빠르게 | *몇 초 안에?* | 1초 안에 |
| 많이 | *몇 건/명?* | 1,000명 동시 |
| 충분한 | *측정 기준은?* | p95 응답시간 200ms 이하 |
| 정확하게 | *오차 허용은?* | 소수점 둘째 자리까지 |
| 사용자 친화적 | *어떤 행동 기준?* | 3 탭 이내 도달 |

### 3.9 문장의 한계 — 그림·표·다이어그램으로 보완

장황한 문장이 길어지면 *Box and Arrow diagram* 이나 표가 효율적입니다.

- **시퀀스가 핵심** → 시퀀스 다이어그램 (Mermaid)
- **다수 컴포넌트 관계** → Box and Arrow diagram
- **다수 필드** → 표
- **상태 전이** → 상태 다이어그램
- **UML 의무화 X** — UML 이 *오히려 부담* 인 경우가 많음. 자유로운 도식이 더 효과적

## 4. 작성 팁 — 자주 빠뜨리는 4 가지

이 4 가지는 *spec-standard.md 에 이미 룰이 있는* 항목이지만, *변환 예제* 만 빠르게 보고 싶을 때 참조합니다.

### 4.1 TBD (To-Be-Determined)

> 결정 못한 항목은 *빈 칸 X, TBD ✓*. 함께 *이유 + 결정자 + 기한* 명시. (spec-standard.md *TBD 처리 규칙*)

| ❌ 빈 칸 | ✅ TBD |
|---------|--------|
| (그냥 비워둠) | `TBD — 결제 PG 선정 미정 (담당: @홍길동, 기한: 2026-06-15, 사유: 수수료 협상 진행 중)` |

### 4.2 N/A vs None

| 표기 | 의미 | 예 |
|------|------|-----|
| **N/A** | *해당 사항 없음* (이 SRS 에 적용되지 않음) | 모바일 전용 앱 SRS 의 *서버 인프라* 섹션 |
| **None** | *해당 사항은 있지만 하지 않음* (의식적 선택) | 다국어 *지원 항목 = None — Phase 1 은 한국어만* |

**둘 다 *이유* 를 함께 적습니다** (30 초 투자로 큰 효과).

### 4.3 비목표 (Will Not Do)

`Will Not Do` 는 *해야 할 일만큼 중요* 합니다. (spec-standard.md *비목표(Will Not Do) 표기 규약*)

| 항목 | 결정 | 이유 |
|------|------|------|
| v1.0 마이그레이션 | 미지원 | v1.0 사용자 3 명, 구조 변경 폭이 커 비용 대비 효과 낮음 |
| 중국어 번체 | 미지원 | 타깃 시장이 *중국 본토 (간체)* — 번체는 Phase 2 |
| HDD 장애 복구 | 미지원 | 현장 방문 지원이 더 효과적, 자동 복구 모듈 개발 비용 과대 |

### 4.4 논의 과정 기록 (Decision Log)

> 한 시간 논쟁의 결론을 *한 줄만* 적지 마세요. 6 개월 뒤 누군가 *같은 논쟁* 을 반복합니다. (spec-standard.md *Decision Log + 핵심 결정 대안 검토 박스*)

핵심 결정마다 다음을 함께 기록:

- *대안* 3 개 비교
- *기각 사유*
- *결정자 + 일시*

## 5. 재사용 · 중복 최소화

### 5.1 한 사실은 한 곳에만 (Single Source of Truth)

> 같은 정보가 *여러 곳에 중복* 되면 *변경 시 한 곳만 수정* 하기 쉬워 *데이터 불일치* 가 발생합니다. (책 §9.8)

**Munto SRS 의 SSOT 룰:**

| 정보 | 정의 위치 | 다른 곳은? |
|------|----------|-----------|
| **DB 스키마** | dev-chain-design 의 DBML | SRS §6.4 는 *논리 구조와 핵심 제약만* 기술, *컬럼 타입·인덱스는 DBML 만* |
| **API 시그니처** | Swagger / OpenAPI | SRS §4.1 · §7 은 *목적·동작·에러 케이스* 만, *parameter/payload 는 Swagger 만* |
| **테스트 케이스** | dev-chain-design 의 TCL | SRS 본문은 *기능 단위만*, *케이스 표는 TCL 만* |
| **UI 디자인** | Figma | SRS §4.2 는 *Figma URL + 핵심 정책만*, *치수·색상은 Figma 만* |
| **변경 이력** | 본 문서 마지막 *변경 이력* + GitHub PR | 본문 안 *수정 이력 박스* 금지 (중복) |
| **비즈니스 규칙** | SRS §6.5 | 7 장 본문은 *기능 동작만*, *규칙 자체는 §6.5* (cross-ref 만) |
| **용어 / 약어** | SRS §1.4 | 본문 첫 등장 시 *정의 → §1.4 참조* |

### 5.2 cross-reference 작성 룰

| 패턴 | 예 |
|------|-----|
| 같은 SRS 내부 | *"자세한 정의는 §1.4 Terms and Abbreviations 참조"* |
| 다른 문서 (같은 폴더) | *"[`spec-standard.md`](./spec-standard.md) §"공통 작성 규칙 → 측정 가능성 원칙"* |
| 외부 도구 (Figma · Swagger · DBML) | *"디자인은 [Figma URL](...) 참조"* + URL 변경 시 모두 업데이트 |
| 일반 외부 문서 | URL + *접근 일자* 명시 (외부 자료는 변경 가능) |

### 5.3 Munto 룰 — Notion · GitHub · Figma 참조

| 시점 | 참조 방향 |
|------|----------|
| 기획 단계 | Notion SRS = 마스터 |
| 설계 단계 (dev-chain-design 시작 후) | DBML / Swagger / TCL = 마스터, Notion 은 *링크 + 요약만* |
| 구현 후 | 코드 = 마스터, SRS / Swagger 는 *코드와 일치하도록 변경 관리 프로세스 적용* |

## 6. Why 를 잘 알아야 한다

> 스펙의 항목들이 *왜 그렇게 적혀야 하는지* 이유를 알아야 합니다. (책 §9.11)

### 6.1 Why - What - How 분리 (spec-standard.md §"Why-What-How 분리" 와 연결)

| 레이어 | SRS 챕터 | 예 |
|--------|---------|-----|
| **Why** | 1.2 Product Scope, 1.1 Purpose | *"가입 전환율을 30% → 45% 로 올린다"* |
| **What** | 2 · 4 · 5 · 6 · 7 장 | *"3 단계 가입 → 1 단계 가입으로 단축"* |
| **How** | (스펙 영역 아님) | dev-chain-{design,backend,frontend,mobile} 산출물 |

### 6.2 Why 가 없을 때 발생하는 문제

| 증상 | 원인 |
|------|------|
| 같은 요구를 *6 개월 뒤 다시 논의* | Decision Log 누락 → §4.4 |
| 신규 입사자가 *왜 이 기능?* 라고 자주 질문 | Purpose 가 *기능 나열* 만 된 SRS |
| *국제화 / 이식성 / 확장성* 요구가 *나중에 발견* | Why 에 *미래 계획* 누락 → §6.3 |

### 6.3 미래 Why 가 현재 What 에 영향

> *1.0 버전에서는 Windows 만 지원하다가 2.0 에서 Linux 도 지원할 계획이라면, 처음부터 이식성 항목에 명확히 기술해야 합니다.* (책 §9.9)

| 미래 계획 | 현재 SRS 에 반영해야 할 항목 |
|-----------|---------------------------|
| 해외 진출 (Phase 2) | §6.10 Internationalization, §6.11 Unicode |
| 다른 OS 지원 (Phase 2) | §6.3.3 Portability, *플랫폼 의존 코드 격리 정책* |
| 사용자 *10 배 확장* (Phase 2) | §5.1 Throughput, §5.2 Concurrent Session, §6.3.5 *Scalability* |
| 외부 API 공개 (Phase 2) | §4.1 System Interface, *공개 API 와 내부 API 분리 정책* |

**Munto AI 룰**: SRS 작성 시 *2.7 Apportioning* 에 Phase 2 계획이 있으면, AI 는 *반드시* 위 표를 참조해 *현재 SRS 에 영향을 주는 항목* 을 짚어야 합니다.

## 7. 훔쳐보기 그만 — 안티 패턴

> Private 함수를 임의로 Public 으로 바꿔서 사용, 라이브러리 내부 함수를 사용하는 등 *설계가 허용하지 않는 참조* 를 *훔쳐보기* 라고 합니다. (책 §9.12)

### 7.1 스펙·설계에서의 훔쳐보기

| 형태 | 결과 |
|------|------|
| 컴포넌트 *내부 모듈* 을 외부에서 직접 호출 | 결합도 증가, 변경 시 다량 장애 |
| 다른 컴포넌트의 *DB 테이블* 을 직접 SELECT | 도메인 경계 무너짐, 마이그레이션 불가 |
| 라이브러리의 *비공개 내부 API* 사용 | 버전 업그레이드 시 깨짐 |
| 순환 참조 | *하나의 변경 = 전체 영향*, 구조 이해 불가 |

### 7.2 SRS 에서 막는 방법

| 방법 | 적용 위치 |
|------|----------|
| *컴포넌트 외부 인터페이스만 §4 · §7 에 정의* (내부는 적지 않음) | §2.2 · §4 |
| *공통 모듈* 을 §2.2 의 컴포넌트와 *다른 형태* 로 명시 | §2.2 |
| *접근 제어 정책* 을 §6.6 Design and Implementation Constraints 에 명시 | §6.6 |
| *DB 접근 책임 컴포넌트* 명시 (다른 컴포넌트는 API 로만) | §6.4 + dev-chain-design |

### 7.3 AI 작성 시 자체 점검

AI 는 SRS 초안을 작성할 때 다음을 *반드시 자체 점검* 합니다.

1. *§7 의 어떤 기능이 다른 컴포넌트의 내부에 직접 접근* 한다고 적었는가?
2. *§4 에 정의되지 않은 인터페이스* 가 §7 에 등장했는가?
3. *순환 의존* 이 §2.2 다이어그램에 그려졌는가?

발견 시 *사용자에게 경고하고 수정 제안* 합니다.

## 8. 인터페이스 정의 3 가지 방법

### 8.1 결정 트리

```
이 인터페이스는 ...
├── 단순 함수 호출 / 내부 API 인가? → §8.2 간단한 정의
├── 외부 시스템 연동 / 결제 / 보안 / 실시간성 / 대용량인가? → §8.3 복잡한 정의 (12 개 속성)
└── REST API / GraphQL 인가? → §8.4 일반적인 정의 (Swagger / OpenAPI 위임)
```

### 8.2 간단한 인터페이스 정의 — 함수 시그니처

내부 컴포넌트 간 *단순 호출* 은 함수 시그니처 + 짧은 주석으로 충분합니다.

```
A모듈 → B모듈
// @param userID : 사용자 ID
// @return 사용자 이름, error 발생 시 "" 리턴
string getUserName(int userID);
```

### 8.3 복잡한 인터페이스 정의 — 12 개 속성

> 같은 인터페이스라도 *하루 1 회 호출* 인지 *초당 1,000 회 호출* 인지에 따라 요구사항이 달라집니다. *함수 호출 정의만으로는 부족* 합니다. (책 §9.14.2)

복잡한 인터페이스(외부 결제, 보안, 실시간 스트림, 대용량)는 다음 *12 개 속성* 을 검토합니다.

| # | 속성 | Munto 적용 예 |
|---|------|--------------|
| 1 | **Priority** (호출 실패 시 영향도) | 결제 = 높음 / 추천 = 중간 / 로그 = 낮음 |
| 2 | **Type** (실시간 vs 배치) | 결제 = 실시간 / 매출 정산 = 일별 배치 |
| 3 | **Data Type** (숫자/문자/혼합/URL) | userID = int, amount = decimal |
| 4 | **Size and format** | amount = 10 자리 이하, currency = 3 자리 ISO 코드 |
| 5 | **Unit** (microsecond/meter/원 등) | amount = KRW, latency = ms |
| 6 | **Range / Enumeration** | status ∈ {PENDING, PAID, REFUNDED, FAILED} |
| 7 | **Accuracy / Precision** | amount = *정수* (소수 미허용), latency = 소수점 둘째 자리 |
| 8 | **Timing** (호출 시기) | 사용자 결제 버튼 클릭 후 *1 초 이내* PG 호출 |
| 9 | **Frequency** (호출 빈도) | 평균 50 RPS, 피크 500 RPS (이벤트 시) |
| 10 | **Volume** (크기) | 요청 payload ≤ 5KB, 응답 ≤ 20KB |
| 11 | **Add / Update 방식** | 중복 요청은 *idempotency key 로 거부*, *덮어쓰기 X* |
| 12 | **Security** | TLS 1.2+, HMAC-SHA256 서명, *PII 마스킹* |

**커뮤니케이션 방법 (별도 5 개 속성)**: 프로토콜·메시지 포맷·흐름 제어·전송 속도·라우팅·보안 — *별도 시스템 연동 시* 만 적용.

### 8.4 일반적인 인터페이스 정의 — Swagger 위임

> 일반적인 인터페이스 정의법은 따로 존재하지 않습니다. 상황에 맞는 방법을 자유롭게 사용합니다. *Swagger 가 REST API 의 사실상 표준* 입니다. (책 §9.14.3)

**Munto REST API 표준 패턴:**

| 작성물 | 위치 | 내용 |
|--------|------|------|
| 엔드포인트 목록 | SRS §4.1 · §7 | *목적·동작·에러 케이스* 만 표 |
| 상세 시그니처 | Swagger / OpenAPI | path / method / parameter / payload / response / status code |
| 비기능 속성 | SRS §5 + Swagger `description` | 응답시간 (§5.3), Throughput (§5.1), Security (§6.2) |
| 변경 이력 | Swagger `info.version` + GitHub PR | major / minor / patch 룰 |

**§8.3 의 12 속성 중 Swagger 가 *자동 커버* 하지 않는 것** (별도 SRS 본문 또는 description 에 명시):

- Priority (호출 실패 영향도)
- Frequency (RPS 목표)
- Timing (외부 이벤트와의 동기)
- Add/Update 방식 (idempotency)
- Security 세부 (서명 알고리즘·키 회전 주기)

## 9. AI 가 스펙 초안을 쓸 때 추가 점검

AI 가 자동 생성한 스펙 초안은 다음 패턴을 *자주 빠뜨립니다*. 사람이 받기 *전에* 자체 점검합니다.

### 9.1 AI 가 자주 빠뜨리는 항목

| # | 빠뜨림 | 점검 방법 |
|---|--------|----------|
| 1 | **에러 케이스** | 모든 §7 기능에 *최소 1 개 에러 시나리오* 있는가? |
| 2 | **Side Effect** | 기능이 *다른 데이터·다른 사용자·다른 시스템* 에 미치는 영향 적었는가? |
| 3 | **Idempotency** | *중복 호출* 시 동작 적었는가? (특히 결제·취소) |
| 4 | **권한 / 인증** | 기능마다 *접근 가능한 User Class* 적었는가? |
| 5 | **데이터 검증** | 입력 *유효 범위·필수 여부·길이 제한* 적었는가? |
| 6 | **빈 / 경계값 처리** | *0 건·null·max 값* 동작 적었는가? |
| 7 | **타임아웃 / 재시도** | 외부 호출에 *타임아웃·재시도 횟수* 적었는가? |
| 8 | **로그 / 감사** | *어떤 로그를 어디에 남기는지* 적었는가? (§6.8 Operations) |
| 9 | **삭제 / 보존 정책** | 데이터 *보존 기간·삭제 방식* 적었는가? (개인정보 법규 영향) |
| 10 | **국제화 / 시간대 / 통화** | *KST 고정인가? UTC 인가?* 다국가 진출 시 영향 |

### 9.2 AI 작성물 사람 검토 체크리스트

> AI 가 *"통과"* 시킨 문장은 *그대로 인수되어 출시* 될 수 있습니다. *통과 = 인수* 원칙. (spec-standard.md *AI 작성물 검증 책임 — "통과 = 인수" 원칙*)

사람 검토자는 AI 초안을 받으면 다음 3 단계를 *순서대로* 거칩니다.

1. **사실성** — 비즈니스 사실이 맞는가? (AI 가 만들어낸 *그럴듯한 거짓* 점검)
2. **완전성** — §9.1 의 10 가지 항목이 모두 점검되었는가?
3. **명확성** — 측정 가능성·모호어·주어·서술식 룰을 따랐는가? (이 문서 §2 · §3)

3 단계 모두 *통과한 후에만* `Approved` 처리.
