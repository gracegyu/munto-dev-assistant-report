## 1. 이 문서가 다루는 것

스펙 작성에는 *두 가지 메타 결정* 이 필요합니다.

| 결정 | 이 문서에서 다룸 | 다른 문서 |
|------|-----------------|----------|
| 어떤 *항목* 을 채울까 | — | `spec-standard.md` (1~9장 고정 구조 + 각 절 작성 가이드) |
| 얼마나 *오래* 쓸까 | §2 | — |
| 얼마나 *자세히* 쓸까 | §3 | — |
| 어떻게 *문장·표·그림* 을 쓸까 | — | `spec-writing-tips.md` |

**AI 에이전트가 이 문서를 읽어야 하는 시점:**

1. 사용자가 *"이 기능, 스펙을 얼마나 적어야 해?"* 라고 물을 때
2. 사용자가 *"일정 부족한데 스펙 건너뛰면 안 되나?"* 라고 물을 때
3. 사용자가 *"이 정도 상세도면 충분해?"* 라고 검토를 요청할 때
4. 새 기능 개발 직전, dev-chain-wbs / dev-chain-design 진입 전에 *문서 분량을 산정* 해야 할 때

## 2. 스펙 작성 시간 가이드

### 2.1 일반 원칙

> 가장 일반적인 방법은 *경험 많은 개발자가 전체 프로젝트 기간을 예측한 후 그 기간의 25~30% 정도를 스펙 작성 기간* 으로 정합니다. (책 §6.4)

이 비율은 *상한선이 아니라 평균치* 입니다. 프로젝트가 복잡할수록 위로, 단순할수록 아래로 조정합니다.

**중요한 비대칭:**

- *프로젝트 기간* 은 잘 작성된 스펙으로 *꽤 높은 정확도로* 산정할 수 있습니다.
- *스펙 작성 기간* 은 정확히 산정할 수 없습니다 — 분석 도중에 알게 되는 *미지의 양* 이 분석 자체를 결정하기 때문입니다.

따라서 *스펙 작성 기간을 못 박지 말고*, 종료 신호(§2.4)가 나타날 때까지 진행합니다.

### 2.2 4 집단 분류 — 어느 위치에 있고 어디로 가야 하나

책 §6.4 는 조직의 성숙도를 4 집단으로 분류합니다.

| 집단 | 분석·설계 비율 | 결과 |
|------|---------------|------|
| **성숙한 개인** | 60~80 % | 코딩 시작 시 거의 모든 결정이 끝나 있어 *재작업 최소* |
| **성숙한 기업** | 25~30 % | 의도적으로 *짧게* 끝내고 *남은 불확실성은 진행 중에 보정*. 비용 효율 최고 |
| **주먹구구식** | 0~5 % | 코딩 일찍 시작 → *재작업의 반복* → 결과적으로 가장 오래 걸림 |
| **주먹구구식 + 문서** | 5~10 % | 문서는 의무로 쓰고 *개발은 주먹구구* → 가장 비효율적. *문서 따로, 개발 따로* |

**Munto 의 위치 — "성숙한 기업" 을 목표로 합니다.**

- 분석 아키텍트 인력이 한정적이라 *한 프로젝트에 너무 오래 매달릴 수 없습니다*.
- 인원·예산 계획이 사전에 정해지므로 *스펙을 빨리 결론* 내야 합니다.
- 남은 불확실성은 *dev-chain (design → backend/frontend/mobile → verify)* 을 통해 단계적으로 보정합니다.

이 위치를 *주먹구구식+문서* 로 떨어뜨리는 가장 흔한 원인은 다음 4 가지입니다.

1. *템플릿 채우기* 에 매몰 — 항목별 *최소 의미 단위* 만 쓰면 됨
2. 설계 단계로 미뤄도 될 결정을 *스펙에 모두 적으려 함*
3. 스펙을 *쓰지 않고 시작* 하고 *완료 후 거꾸로 작성*
4. AI 가 *진짜 검토 없이 통과시킨* 초안을 그대로 인수

> AI 안티 패턴: 사용자가 *"스펙 작성 시간 부족"* 이라고 하면, *"항목 줄여드릴게요"* 가 아니라 *"기능 범위를 줄이거나 다음 버전으로 미루세요"* 라고 답해야 합니다. *스펙을 부실하게 만드는 것은 일정 단축이 아닙니다*.

### 2.3 Munto 적용 — 산출물별 권장 시간

문토 dev-chain 의 산출물별 권장 작성 시간입니다. *전체 개발 일정의 25~30%* 룰을 산출물 단위로 분배한 결과입니다.

| 산출물 | 권장 작성 시간 | 적용 기준 |
|--------|---------------|----------|
| **Engineering One Pager** | 0.5 ~ 1 일 | 새 기능/실험 착수 *직전*. 경영진/이해관계자 승인용 |
| **SRS — 소규모** | 1 ~ 2 일 | 단일 화면, 1~2 API, 외부 연동 없음 |
| **SRS — 중규모** | 3 ~ 5 일 | 다중 화면, 5~15 API, 1~2 개 외부 연동 (예: 결제, 푸시) |
| **SRS — 대규모** | 1 ~ 2 주 | 신규 도메인, 다수 화면, 15+ API, 3+ 개 외부 연동, *2.2 컴포넌트 분리 필요* |
| **WBS (dev-chain-wbs)** | SRS 작성의 10~20% | SRS 가 안정된 직후 |
| **DBML / Swagger / TCL (dev-chain-design)** | 전체 개발 일정의 5~10% | SRS 확정 직후 *병렬 산출* |

**대규모 SRS 가 *2 주 이상* 걸리면 신호입니다:**

- 기능 범위가 *너무 큼* → Phase 분할 (§2.7 Apportioning) 권장
- *외부 의존성이 미해결* → 의존성 정리부터
- *분석 아키텍트가 1 명* → 책 §6.12 *큰 프로젝트 협업 분석법* 필요

### 2.4 스펙 작성을 종료할 신호 5 가지

> *사용자가 요구사항을 더 이상 생각해 내지 못하거나, 이전의 리뷰 시간에 했던 얘기를 자꾸 반복하기 시작하면 끝낼 때가 된 것이다.* (책 §6.5)

AI 가 스펙 검토 중 다음 신호 5 개 중 *3 개 이상* 이 감지되면 *"이제 종료 시점"* 이라고 사용자에게 알립니다.

| # | 신호 | 의미 |
|---|------|------|
| 1 | 사용자가 새 요구사항을 *3 번 이상 같은 말로 반복* | 새 정보 없음. 인지된 범위 다 나옴 |
| 2 | 신규 요구가 *프로젝트 목표·범위에서 벗어남* | 다음 버전으로 미룰 시점 |
| 3 | 신규 요구의 *우선순위가 모두 P3 이하* | 핵심은 다 잡힘 |
| 4 | 리뷰에서 *문장 표현·오타* 만 지적됨 | 내용은 끝났고 *형식 마무리* 단계 |
| 5 | 사용자가 *"일단 이 정도로 가자"* 라고 명시 | 종료 합의 |

**종료 후에도 *변경은 변경 관리 프로세스* 로 처리합니다** — 책 §6.9, spec-standard.md §8 (Change Management Process) 참조.

## 3. 스펙 상세도 가이드

### 3.1 절대 법칙은 없다

> 스펙에서 어느 정도 깊이까지 정보를 다룰지는 *프로젝트의 규모·성격·특성에 따라 다릅니다*. 절대 법칙은 없습니다. (책 §2.8 + §6.5)

AI 에이전트는 *"무조건 자세히"* 또는 *"무조건 간결히"* 가 아니라, **5 개 축으로 자체 판정** 한 뒤 사용자에게 *권장 상세도* 와 *근거* 를 제시합니다.

### 3.2 Case A/B/C — 책의 분류

책 §6.5 는 3 가지 케이스를 제시합니다.

| 케이스 | 조건 | 권장 상세도 |
|--------|------|------------|
| **Case A — 저상세** | 요구분석·설계·구현 인력이 *모두 동일*, 유지보수 프로젝트, 비슷한 프로젝트를 많이 해봄 | *간단히* |
| **Case B — 중상세** | 각 단계 인력이 *다름*, 처음 해보는 종류, 불확실한 기술 검증 필요 | *중간* |
| **Case C — 고상세** | *외주*, *생명을 다루는 의료/안전 제품*, *지리적으로 분산*, *외국인 개발자 포함* | *상세히* |

### 3.3 5 개 축 결정 표 (AI 가 적용)

각 축을 *간단히 ↔ 상세히* 사이에서 평가합니다.

| 축 | 간단히 | 상세히 |
|----|-------|--------|
| **프로젝트 난이도** | 쉬운 — 비슷한 패턴 다수 | 어려운 — 새로운 도메인·알고리즘·시장 |
| **커뮤니케이션 용이성** | 구현 기간에 분석 아키텍트에게 *수시로 물어볼 수 있음* | 구현 기간에 분석 아키텍트를 *만나기 어려움* (휴가·퇴사·다른 프로젝트 이동) |
| **개발팀 경험** | 비슷한 프로젝트를 *많이 해봄* | *신규 개발자가 많음* |
| **참고 제품 유무** | 비슷한 제품/모듈이 *사내·외에 많음* | *참고할 만한 것이 없음* |
| **외주 여부** | *내부 경험 많은 개발팀* | *외주* 또는 *지리적 분산팀* |

**AI 판정 룰:**

- *상세히* 점수가 *3 개 이상* 이면 → **Case B 또는 C — 상세 작성**
- *상세히* 점수가 *0~1 개* 이면 → **Case A — 간결 작성 (단, 최소 항목은 모두 채움)**
- *상세히* 점수가 *2 개* 이면 → **Case B — 중간**

### 3.4 Munto 도메인별 권장 상세도

문토 도메인별로 *기본값* 을 제시합니다. 위 5 개 축 결과로 조정합니다.

| 도메인 | 기본 상세도 | 근거 |
|--------|------------|------|
| **앱 (UI 위주, 신규 화면)** | Case A (간결) | dating-mobile / munto-mobile 에 *유사 패턴 다수*, BLoC/Riverpod 컨벤션 정착 |
| **앱 (외부 SDK·결제·푸시·홈위젯)** | Case B (중간) | 외부 의존성 검증 필요 |
| **웹 (UI 위주, 신규 화면)** | Case A (간결) | munto-frontend MVVM 패턴 정착 |
| **백엔드 (CRUD 수준 신규 API)** | Case A (간결) | NestJS+Prisma 패턴 정착, dev-chain-design 산출물로 보완 |
| **백엔드 (도메인 로직 — 정산, 매칭, 추천 등)** | Case B (중간) | *비즈니스 규칙* 이 외부 시스템·법규와 얽힘 |
| **AI / 추천 / 매칭** | Case C (상세) | *정확도·재현율·실패 시 동작* 등 *품질 특성* 까지 명시 필요 |
| **결제 / 정산 / 환불** | Case C (상세) | *금전·법규* 직결 — 외부 PG, 세무, 사기 방지까지 |
| **외부 연동 (Meta Ads, 카카오, 애플/구글)** | Case C (상세) | *외부 정책 변경 리스크*, 인증·심사 절차 명시 |

> 이 표는 *기본값* 입니다. 5 개 축 평가 결과가 다르면 *그쪽을 우선* 합니다. 예: "신규 화면이지만 외주" → Case C.

### 3.5 안티 패턴 — *"문서만 보고 개발 가능"* 강박

> *"문서만 보고 개발할 수 있을 정도로 스펙을 자세히 적는 것이 좋다"* — 이것은 *잘못된 통념 6* 입니다. (책 §3.2)

**경험 많은 개발자가 투입되는 프로젝트에서는 *오히려 간결하게* 적어야 빨리 끝납니다.** *모든 챕터를 다 자세히 적으면 중복이 많이 발생* 하고, 변경 시 *여러 곳을 동기화* 해야 하므로 유지보수 비용이 폭증합니다.

| 잘못된 시그널 (AI 가 자체 점검) | 올바른 시그널 |
|--------------------------------|--------------|
| *"외주에 줄 수도 있으니까 무조건 상세히"* | *"외주가 확정이면 상세히, 아니면 케이스에 맞춰"* |
| *"신입이 볼 수도 있으니까 무조건 상세히"* | *"투입 인력이 정해지면 그 인력에 맞춰"* |
| *"나중에 잊을 수도 있으니까 무조건 상세히"* | *"잊을 만한 *핵심 결정* 만 Decision Log 에 남기기"* |
| *"AI 가 자동으로 잘 쓰니까 상세히"* | *"AI 가 잘 쓸수록 *사람 검토 부담* 이 증가 — 검토 가능한 분량 으로"* |

### 3.6 상세도 부족이 더 위험한 경우

반대로 *Case A 라도 반드시 상세히 적어야 하는 항목* 이 있습니다.

| 항목 | 이유 |
|------|------|
| **외부 인터페이스 (§4 · §7)** | 외부 시스템과 *반드시 일치* 해야 함 — 빠지면 통합 실패 |
| **비기능 요구사항 (§5 · §6)** | *측정 가능한 숫자* 가 빠지면 *합격/불합격 판정 불가* |
| **DB 논리 스키마 (§6.4 + DBML)** | 도메인 모델은 *교체 비용이 큼* |
| **법·인증·표준 제약 (§6.6)** | 빠지면 *재작업 + 출시 지연 + 법적 리스크* |
| **에러 케이스 / Side Effect (§7)** | *해피 패스만 적힌 스펙은 절반밖에 안 적은 것* |

### 3.7 Munto 신설 — *AI 가 검토할 수 있는 분량* 룰

AI 가 스펙을 작성·검토할 때는 *추가 제약* 이 있습니다.

- **한 SRS = 사람이 한 자리에 앉아 끝까지 읽을 수 있는 분량.** Notion 한 페이지 기준 *2,000~6,000 단어* 권장.
- 이 범위를 넘으면:
  - Phase 분할 (§2.7 Apportioning) 로 *Phase 1 / Phase 2* 별도 SRS
  - 또는 *컴포넌트별 SRS 분리* (예: 앱용 / 백엔드용)
- AI 가 *6,000 단어 초과 SRS* 를 작성하려 하면 사용자에게 *"분할을 권장합니다"* 라고 *먼저* 알립니다.

## 4. 자주 묻는 질문 (FAQ)

### Q1. SRS / One Pager / WBS / Notion 메모 — 어디까지 써야 하나?

| 상황 | 권장 |
|------|------|
| *아이디어 검증 / 경영진 승인 받기 전* | One Pager 만 |
| *경영진 승인 후, 1~2 일 작업* | One Pager + Notion 메모. SRS 생략 가능 |
| *경영진 승인 후, 3 일 이상 작업* | One Pager + SRS + WBS + dev-chain-design |
| *외부 의존·결제·AI·법규 관련* | 위 + 상세 SRS (Case C) |

### Q2. 일정이 부족한데 스펙을 건너뛰어도 되나?

**아니요.** 책 §3.2 *잘못된 통념 1·2·6·7* 모두 이 질문의 변형입니다. 답은 정해져 있습니다.

- *"스펙을 써 보지도 않고 일정이 부족하다는 것을 아는 것이 모순"* — 일정 부족 자체가 *스펙 작성으로 검증* 되어야 함.
- 진짜 스펙을 쓸 시간조차 없다면 *프로젝트 시작을 미루는 편* 이 낫습니다.
- 그래도 진행해야 한다면 *Phase 1 만 상세 스펙, Phase 2 는 추후* 로 분할하세요.

### Q3. 스펙이 *너무 자세해* 보이는데 줄여도 되나?

§3.6 의 *반드시 상세히 적어야 하는 항목* 5 가지가 누락 없이 충분히 적혀 있다면, 나머지는 *Case A/B/C 결정 후 다시 평가* 하세요. 검토 부담이 크면 보통 다음 중 하나가 문제입니다.

1. 같은 내용이 *여러 곳에 중복* (→ spec-writing-tips.md §5 *중복 최소화* 적용)
2. *Why·What·How* 가 섞임 (→ Why·What 만 남기고 How 는 dev-chain-design 으로)
3. *예시·근거가 본문에 박힘* (→ Appendix 로 이동)

### Q4. 스펙 작성 후에도 요구사항이 계속 바뀌는데?

> *99% 의 소프트웨어 프로젝트는 분석 기간은 물론이고 설계·구현 중에도 요구사항이 계속 바뀝니다.* (책 §3.2 잘못된 통념 7)

이건 *스펙을 안 적어야 할 이유가 아니라, 변경 관리 프로세스가 필요한 이유* 입니다. spec-standard.md §8 (Change Management Process) 의 *CCB·영향 분석·Decision Log* 룰을 적용하세요.
